1. 目标与唯一基线

本计划验证 Prototype 是否能够安全闭环以下核心业务：账户绑定、结构化即时下单、订单与礼物资金预留、透明匹配、陪玩工作台、双方就绪、取消影响预览、客服接管、全人工送礼核对、两种互斥的一级返佣、消费与收益时间线、四级后台权限、Discord /bot-config 基础配置、八项运营指标以及故障恢复。

唯一上位规格 《Discord陪玩业务Bot最小原型设计开发文档.html》及《最新需求统一基线》v0.7。出现解释差异时，以 v0.7 对 P0、资金预留、双方就绪、返佣保密和运营指标的最新约束为准。CSV 通过 PL-01..PL-06、BASELINE-* 和 SPEC-x.y 追踪来源。

6. 测试环境与数据

6.1 环境要求

  • 独立 Discord 测试服务器、测试 Bot App、测试 Role 和固定频道，不复用生产 ID。
  • 独立 API、PostgreSQL 和 Job Worker；时间相关测试支持 fake clock 或可控 Job 触发。
  • 第三方沙箱或行为等价 Mock，支持余额不足、超时、未知结果、重复与乱序 Webhook。
  • L1-L4 各至少一个员工账号；Fixtures 覆盖同一员工发起并执行，不要求准备独立复核人员。

6.2 Fixture 使用规则

test-fixtures.json 提供 128 个稳定的 FX-* fixture key；实际内部实体使用确定性 UUIDv5，Discord 用户、Guild、频道、消息和 Role 使用确定性 18 位纯数字测试 Snowflake，外部 provider ID 使用明确的测试格式。CSV 的“前置数据”字段通过 fixtureIndex 和 JSON Pointer 装载数据。

  • 每条自动化用例在独立事务、独立 schema 或唯一 test_run_id 下执行。
  • 并发场景必须使用真实数据库事务，不用进程内锁模拟。
  • 所有 Adjustment 的 amountMinor 使用非负绝对值，方向只由稳定的 type 表达。
  • Fixtures 不含真实 Token、Key、个人资料或支付凭证。
  • 测试结束可清理普通临时数据；财务与审计测试通过隔离数据库重置，不调用产品硬删除接口。

7. 执行与证据

7.1 推荐批次

  1. Smoke：绑定、完整确认、订单预留、匹配、接单、双方就绪、完成、消费和收益。
  2. Money：三段余额、并发预留、捕获/释放、取消预览、礼物审批、返佣和收益 Adjustment。
  3. Access：L1-L4、Role 升降级、/bot-config 权限与 ephemeral 隔离、Actor 伪造、跨用户访问和无硬删除。
  4. Resilience：Webhook 重复/乱序、第三方超时、Outbox、Bot 重启、面板与数据库恢复。
  5. UAT：陪玩工作台、客服暂停/恢复、返佣保密、四级 Dashboard、八项指标和备份恢复演练。

7.2 每条用例证据

层级最低证据

恢复演练执行时间线、故障注入点、恢复结果、重复副作用计数、恢复后健康检查

7.3 缺陷等级

  • Blocker 重复扣退款、越权、数据泄露、多人接单、财务半完成、不可恢复或不可变记录被修改。
  • High 核心流程无法闭环、错误状态迁移、超时未进入客服、Role 撤权不及时、关键审计缺失。
  • Medium 非阻断展示、文案、排序或人工可恢复的同步问题。

8. 准入、退出与放行

8.1 开始验收前

  • 主规格版本冻结，API 契约和数据库迁移可部署。
  • 测试 Bot、Dashboard、API、Worker、PostgreSQL 和第三方 Mock/沙箱均可访问。
  • 服务双价格、礼物、频道、Role、阈值和 Fixtures 已导入。
  • 日志、指标、审计和 Job 重试页面可用于取证。

8.2 Prototype P0 放行条件

  • 108 条 P0_BLOCKER 全部通过，不能豁免。
  • 44 条 P0_HIGH 全部执行；未通过项必须有明确修复版本，不允许影响核心闭环、权限隔离或资金一致性。
  • PL-01..PL-06 及对应扩展场景全部通过；提交前确认、预留、匹配、工作台、双方就绪和取消预览不得降级为客户端本地规则。
  • ¥499/500/501、¥1,999/2,000/2,001/5,000，以及权限累积继承、同一人发起并执行和高额仅 L4 直接执行全部通过。
  • 并发订单/礼物不超支、重复捕获、并发抢单、重复/乱序 Webhook、退款交易与原扣款聚合状态分离、返佣零泄露、Role 降权、无硬删除、审计和恢复演练全部通过。
  • 八项指标均可由 fixture 源事实复算；时间线在 L1-L4 下按权限正确脱敏。
  • AT-CFG-001..010 全部通过；配置必须先预览并取得绑定变更的短期 Token 再确认，API 拒绝绕过预览的直接保存；成功后下一次业务动作立即使用新配置，版本冲突不覆盖新值，变更可审计，重启从 API 重载且配置响应仅操作者可见。
  • 测试环境完成一次 Bot 重启、面板重建、Outbox 恢复和数据库备份恢复。
  • 产品、客服运营、技术负责人共同签署 UAT 结果。

8.3 固定测试假设

  • 订单确认提交时创建资金预留；完成时捕获；用户取消、待派单预留 30 分钟到期、拒绝或失败时释放；5 分钟派单轮次超时保持；争议保持，部分结案按决议捕获并释放剩余。
  • 礼物请求创建时立即预留；批准时捕获已有预留；拒绝、过期或撤回时释放。客服核对信息，不人工计算可用余额。
  • 余额字段固定为 providerBalanceMinor、reservedMinor、availableMinor、currency 和 fetchedAt；可用余额等于第三方余额减有效预留合计。
  • 阈值使用包含边界：L2 礼物 ≤ ¥2,000，L2 退款 ≤ ¥500；高于 L2 上限且低于 ¥5,000 的操作由 L3+ 执行；单笔 ≥ ¥5,000 仅 L4 可直接执行，允许同一人发起并执行，不要求不同人员复核。
  • 返佣采用 basis points；不足一个 minor unit 时向下取整，避免浮点运算。
  • PROMOTER_FIRST_PURCHASE 与 PLAYER_LIFETIME 互斥；被推荐用户端不显示任何受益人、关系类型、比例、金额或状态。
  • 订单从 ACCEPTED 仅在用户与陪玩均就绪后进入 IN_SERVICE；单方就绪和就绪超时均不开始计费。
  • Role 的最高映射值只是 requested level；内部 staff_accounts.level 是最终授权事实来源。
  • /bot-config 的最终授权由 API 基于 staff_account 和 bot_config.read、bot_config.operational.manage、bot_config.security.manage 判定；不制作独立网页或可点击配置 Demo。
  • 交易类型固定为 ORDER_CHARGE、GIFT_CHARGE、ORDER_REFUND、GIFT_REFUND；Webhook 使用 DEBIT_UPDATED / REFUND_UPDATED，事件类型与交易状态分开保存。
  • 退款交易过程状态使用 UNKNOWN / PENDING / SUCCEEDED / FAILED；原扣款是否已部分或全部退款使用独立聚合状态断言。
  • PlayerEarning 和 Commission 每个业务来源只保留一条主记录；退款或纠错追加 PlayerEarningAdjustment / CommissionAdjustment，不创建第二条收益或返佣主记录。
  • 所有财务、订单事件和审计数据只追加；纠错使用 Adjustment 或其他追加更正记录。
