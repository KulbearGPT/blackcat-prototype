共同约束

主题必须遵守
业务事实来源订单、预留、礼物、返佣、收益、审批与权限由统一 API 和 PostgreSQL 保存；Bot 与 Dashboard 不复制业务规则。
资金边界第三方保存账户、充值、真实余额并执行资金动作；自有系统保存订单、预留、消费、退款镜像和可追溯流水。
余额口径availableMinor = providerBalanceMinor - reservedMinor；预留是记录合计，不是可手工编辑字段。
权限L1 基础客服、L2 客服主管、L3 运营负责人、L4 管理员/所有者；权限按 L1<L2<L3<L4 累积继承，高级别自动拥有低级全部执行权限。业务操作采用最低等级直接执行，同一人可发起并执行；高额 ≥500000 必须 L4 但不强制不同人员复核；保留原因、step-up/MFA 和不可篡改审计。API 是最终授权点。
返佣隐私PROMOTER_FIRST_PURCHASE 与 PLAYER_LIFETIME 互斥；被推荐用户永远不得获知受益关系。
数据删除订单、资金、消费、收益、礼物、返佣、退款与审计不硬删除；使用释放、冲正、禁用、归档或追加更正。
P0 非目标不做指定陪玩、预约排班、完整 BI/导出、多 Server/多渠道、原生移动端、白标、多租户、VIP 或营销系统。

进入开发门槛

UI 可追踪
每个 P0 按钮、Modal 和 Dashboard 写操作都有明确状态、反馈与 OpenAPI operationId。
接口可生成
openapi.yaml 可被 OpenAPI 3.1 工具解析，并包含认证、权限、错误、幂等、预留和示例。
数据可迁移
schema.prisma 通过格式与校验；不可变记录、唯一约束、预留生命周期和状态枚举明确。
配置可启动
业务配置通过 JSON Schema；测试 Guild、频道、Role、服务、预留策略和返佣计划已准备。
第三方可验证
取得沙箱凭证和供应商文档；balance/hold/capture/release/refund 能力与降级策略有证据。
故事可验收
Backlog 中每个 P0 Story 映射主规格与测试用例；P0 blocker 进入 CI 或明确 UAT 清单。
范围不倒灌
P1-01—P1-06 和 Nice to Have 不进入 P0 实现，除非重新审批范围并同步所有正式交付物。
