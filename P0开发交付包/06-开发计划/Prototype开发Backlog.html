1. 交付基线

本 Backlog 不重新解释产品方向，只把主规格中的 P0 转为可排期、可验收的开发单元。任何冲突均以上位规格为准；新增 operationId 是对主规格 API 路径的稳定命名，不扩大业务范围。

统一边界：Sapphire Bot 和 React Dashboard 只调用统一业务 API；只有 API 访问 PostgreSQL。第三方系统负责用户/余额/扣款/退款；本地保存用户映射、订单、派单、礼物、一级返佣、消费、陪玩收益、客服任务、审批和审计。

4固定管理级别 L1–L4
  • 金额统一使用 minor units + currency；余额响应固定为 providerBalanceMinor、reservedMinor、availableMinor、currency、fetchedAt。
  • 订单只做即时单；用户和陪玩各最多一个活跃订单；一单一名陪玩。
  • 统一权限口径：权限按 L1<L2<L3<L4 累积继承，高级别自动拥有低级全部执行权限；业务操作采用最低等级直接执行，同一人可发起并执行。
  • L1 不执行扣款、退款、取消已接单订单、封禁、配置修改等资金或破坏性动作；退款 ≤50000 由 L2、50001–499999 由 L3、≥500000 由 L4 执行，礼物 ≤200000 由 L2、200001–499999 由 L3、≥500000 由 L4 执行。高额 ≥500000 必须 L4，但不强制不同人员复核；操作仍须保留原因、按策略完成 step-up/MFA，并写入不可篡改审计。
  • 订单、外部交易、消费、退款、订单事件和审计只追加；收益/返佣冲正仅新增 PlayerEarningAdjustment/CommissionAdjustment，不提供硬删除。

2. 稳定需求 ID 与启动优先级

以下 ID 是排期、OpenAPI、测试和验收证据的稳定追踪键。所有条目当前 delivery_status=PLANNED、target_release=P0；具体 Story 映射显示在 Backlog 行内，并与同目录 CSV 一致。

4. 关键路径与并行轨道

M0 工程/数据/鉴权→M1 确认/预留→M2 透明派单/双方准备→M3 礼物/返佣→M4 权限/时间线/指标→M5 E2E/沙箱/恢复
  • API/Data 轨：M0-US-02/03/04/05 → M1-US-01/02/03/05 → M2 领域用例 → M3 财务域 → M4 安全与审批。
  • Sapphire Bot 轨：M0-US-01/03 → M1-US-04/06 → M2-US-01/02/03/04 → M3-US-01/03；Bot 只负责 Discord 适配和渲染。
  • Dashboard 轨：M4-US-01 可在 M0 鉴权契约稳定后用 Mock API 提前启动；业务页面等待相应 M2/M3 读写 API。
  • QA/DevOps 轨：从 M0 建 smoke/contract tests，每个里程碑加入回归；M5 负责整体验收而不是第一次开始测试。
  • 可并行原则：同一 operationId 的 OpenAPI schema 和错误码先冻结，Bot/Dashboard 再并行开发；状态机、金额和权限规则不得由客户端先行自定义。

5. 2–4 人小团队 Sprint 编排

以下是依赖顺序，不代表固定日期。每个 Sprint 只承诺团队实际可用产能的 75%–80%，其余用于第三方波动、Discord 权限/交互验证、评审返工和线上化准备。

Sprint 0：M0 基线完成工程、Schema、Actor/幂等、Payment Mock、Outbox。退出前必须有一条 Bot→API→DB 的可审计测试调用。
Sprint 1：M1 下单与预留API/Data 处理目录、绑定、草稿、完整确认和预留并发；Bot 并行完成公共入口、频道和面板。
Sprint 2：M2 订单闭环打通透明派单、唯一接单、陪玩工作台、双方准备和正常完成，再补取消预览、暂停接管与异常结案。
Sprint 3：M3 + M4 外壳完成礼物预留/审批/捕获、消费、收益和两种保密返佣；并行完成 OAuth、Dashboard 壳和只读页面 Mock。
Sprint 4：M4 权限运营面统一工作台、四级权限、审批、Role 同步、交易时间线、八指标、审计和失败任务进入跨角色 E2E。
Sprint 5：M5 验收与上线化真实 Provider 沙箱、全链路回归、备份恢复、Bot/Worker 重启、权限安全检查、业务 UAT 和发布门禁。

建议分工

  • 2 人：每个 10 工作日 Sprint 净产能约 15–16 人日，覆盖 168–273 人日约需 11–19 个 Sprint；一人主 API/Data/Payment，一人主 Sapphire/Dashboard。
  • 3 人：每 Sprint 净产能约 22–24 人日，约需 8–13 个 Sprint；API/Data、Sapphire/Integration、Dashboard/QA 各一主责。
  • 4 人：每 Sprint 净产能约 30–32 人日，约需 6–10 个 Sprint；Platform/Payment、Domain API/Data、Sapphire、Dashboard/QA 四轨。
风险缓冲：总体估时上增加 20%–25%；真实 Provider、Webhook 和 Discord 权限/Presence 相关 Story 使用 25%–35% 专项缓冲。缓冲不得预先装入 P1 功能。

6. Definition of Ready 与 Definition of Done

Story 可进入开发

  • 依赖 Story 已完成，或团队提供了版本固定的 Mock/契约。
  • 涉及的 operationId、请求/响应、错误码、Actor 类型和幂等键规则已经明确。
  • 涉及金额时已声明 minor units、currency、上下限和审批阈值。
  • 涉及状态时已列出合法迁移、并发冲突和失败后的业务状态。
  • 涉及 Discord/Dashboard 时已确认组件、权限、加载、空、错误、无权限和陈旧数据表现。

所有 Story 的共同 DoD

  • 本 Story 的验收条件逐项通过，且证据关联 Story ID。
  • API operationId 与 OpenAPI 保持一致；Bot/Dashboard 使用生成或统一 API Client，不复制业务规则。
  • 单元、集成和适用的 E2E 测试进入 CI；金额、权限、并发、重复提交和失败分支纳入测试。
  • 数据库变更包含可重复迁移、约束测试和种子数据影响；不可变记录没有硬删除路径。
  • 写操作接入 Actor Context、幂等、审计和结构化日志；日志不包含 Token、绑定码、第三方密钥或支付敏感数据。
  • 配置、运行手册、错误码和可观测指标随实现更新；代码经至少一名非作者评审。
