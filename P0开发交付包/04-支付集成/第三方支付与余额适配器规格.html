第三方支付与余额适配器规格

以供应商原生 hold 为首选、以经批准的本地 FundReservation 为回退，统一定义能力发现、预留、捕获、释放、退款、过期、对账与 Webhook 边界。

1. 定位与边界

本文件是计划中的 P0 开发契约，不表示真实供应商已经接通。Sapphire Bot 与 Dashboard 只调用统一业务 API；第三方资金能力只能由 API 内部的 Payment Adapter 使用。

核心规则：第三方继续负责用户账户事实、充值、支付、退款和真实余额；自有系统负责订单、FundReservation、消费、返佣、陪玩收益、权限、审计与运营视图。

第三方事实

外部用户、providerBalanceMinor、原生 hold、实际 debit、refund 与供应商状态。

业务事实

FundReservation、reservedMinor、availableMinor、订单/礼物状态、消费与后续结算。

适配器职责

能力发现、协议映射、认证、签名、超时、状态规范化、查询恢复和 Webhook 验证。

P0 不包含

不自建钱包，不做汇率换算，不加入预约排期，也不加入陪玩试音或用户选陪玩。

机器可读接口、状态、Mock 和测试 ID 以 adapter-contract.yaml 为准。

2. 能力发现与模式选择

discoverCapabilities 返回观察到的能力，不能自行开启生产路径。L4 批准的版本化集成配置依据正式文档、沙箱结果和接入清单，在两种模式中选择一种。

模式启用条件预留事实捕获动作
PROVIDER_NATIVE_HOLDcreate/capture/release/get、写幂等、按键查询和可用 TTL 全部有阻断证据。本地 FundReservation 镜像供应商 hold，并保存 provider hold ref 与有效期。完成或批准时调用 captureHold。
LOCAL_RESERVATION_FALLBACK原生 hold 不完整或 TTL 不足；回退 debit 与查询能力已证明，并经 L4 明确批准。FundReservation 是本地并发控制事实，不声称第三方已冻结资金。完成或批准时复查余额后调用 createReservationDebit。
  • 每个 FundReservation 创建时快照 funding mode；生命周期内不得在两种模式之间切换。
  • 供应商能力降级时停止新预留并告警；已存在预留沿原模式查询、释放或人工处置。
  • 若原生 hold 的最大 TTL 不能覆盖配置的业务窗口与安全余量，创建预留前就选择本地回退；不得等 hold 过期后偷偷改成 debit。

4. 稳定接口

interface PaymentProviderAdapter {
  discoverCapabilities(): Promise<ProviderCapabilities>;
  resolveUser(input: ResolveUserInput): Promise<ResolvedUser>;
  getProviderBalance(externalUserId: string): Promise<ProviderBalanceSnapshot>;
  createHold(input: CreateHoldInput): Promise<Hold>;
  captureHold(input: CaptureHoldInput): Promise<Hold>;
  releaseHold(input: ReleaseHoldInput): Promise<Hold>;
  getHold(input: HoldLookup): Promise<Hold>;
  createReservationDebit(input: ReservationDebitInput): Promise<Transaction>;
  createRefund(input: RefundInput): Promise<Transaction>;
  getTransaction(input: TransactionLookup): Promise<Transaction>;
  verifyWebhook(input: RawWebhookEnvelope): Promise<ProviderEvent>;
}
操作用途关键约束
discoverCapabilities返回 native hold、fallback debit、refund、Webhook 与 TTL 能力。只报告能力，不替代接入批准。
resolveUser解析外部用户绑定。一次性凭证用后即弃，不进日志。
getProviderBalance返回供应商余额快照。字段是 providerBalanceMinor，不是 available；reserved/available 由业务 API 计算。
createHold原生模式创建 hold。需要 FundReservation 绑定、版本、业务来源、业务 UUID 与 expiresAt。
captureHold捕获全部或部分 hold。只用于订单完成、礼物批准或批准的部分结案。
releaseHold释放全部或剩余 hold。捕获前取消/拒绝/撤回/失败/过期使用 release，不使用 refund。
getHold按 hold ref 或幂等键查询。处理超时、PENDING、UNKNOWN、重启与回调补偿。
createReservationDebit本地回退模式捕获预留。fundReservationId 与 fundReservationVersion 都必填。
createRefund捕获后的全额或部分退款。成功及未决退款累计不超已捕获金额。
getTransaction查询 fallback debit 或 refund。写超时后先查，不生成新键。
verifyWebhook验签并规范化 hold/debit/refund 事件。针对原始 bytes 先验签再解析。

5. 余额与预留语义

唯一命名：适配器输出 providerBalanceMinor。统一业务 API 的用户余额响应固定为 providerBalanceMinor、reservedMinor、availableMinor、currency、fetchedAt。
reservedMinor  = Σ 当前有效 FundReservation.amountMinor
availableMinor = providerBalanceMinor - reservedMinor
  • reservedMinor 是有效记录合计，不是可手工编辑的字段。
  • availableMinor 由业务 API 计算；不得把第三方字段直接透传并改名为 available。
  • 本地回退时，第三方站外消费可能令 availableMinor 为负。API 保留该有符号结果，阻止新预留并触发缺口告警。
  • 用户提交前看到的余额摘要是判断依据，不是对稍后捕获成功的供应商承诺。

6. FundReservation 生命周期

阶段原生 hold本地回退共同业务规则
创建创建本地记录并以固定键调用 createHold。锁用户资金域，读取新鲜余额并原子创建本地记录。绑定唯一 ORDER 或 GIFT、幂等键、mode、expiresAt 与 version。
有效供应商状态 ACTIVE，本地镜像持续对账。本地记录占用 reservedMinor，第三方并未冻结。重复请求返回同一记录；不得另建第二预留。
捕获captureHold 成功后写消费事实。锁记录、校验 version、复查余额，再 createReservationDebit。PENDING/UNKNOWN 锁住业务终态并进入对账。
释放releaseHold 成功或确认已过期后释放。本地事务更新 RELEASED 并递增 version，无第三方 release。捕获前取消、拒绝、撤回、失败和过期走此路径。
退款已捕获金额只能通过 createRefund 冲正；原捕获、消费、返佣和收益记录不覆盖不删除。退款与 Adjustment 只追加。

部分结案先捕获批准金额，再释放剩余金额；两者使用不同的稳定幂等键。每个 Hold 响应必须返回 capturedAmount、releasedAmount 与 remainingAmount，三者合计等于原预留。

7. 订单与礼物映射

7.1 订单

确认提交→创建预留→派单/接单→用户与陪玩分别就绪→IN_SERVICE→完成时捕获
  • 订单在用户确认提交时 reserve，不等待服务开始。
  • 订单从 ACCEPTED 只有在用户与陪玩双方都 readiness 后才进入 IN_SERVICE；readiness 不触发 capture。
  • 订单完成时 capture。捕获成功后才创建消费、陪玩收益与适用的返佣事实。
  • 捕获前取消、无人接单、拒绝或失败 release；捕获后取消或争议按批准金额 refund。

7.2 礼物

创建请求→立即预留→L1 核对→按金额审批→批准时捕获→Outbox 播报
  • 礼物在用户创建请求时 reserve，不能等客服批准后再抢余额。
  • 客服核对礼物品类、目标和真实意愿，不人工计算余额；API 捕获已有预留。
  • 拒绝、过期或撤回在捕获前 release；捕获后纠错使用 refund。
  • 播报失败只重试 Outbox，不重复 capture 或 debit。
捕获后的业务事实：PROMOTER_FIRST_PURCHASE 与 PLAYER_LIFETIME 只能依据成功消费事实结算；退款或结案用追加 Adjustment 冲正。

8. 本地回退防线

  1. 业务动作锁定目标 FundReservation，并确认 mode 为 LOCAL_RESERVATION_FALLBACK。
  2. 以 id + expected version 原子更新；传入 fundReservationVersion 与当前版本不符时返回 RESERVATION_CONFLICT。
  3. 调用 getProviderBalance 取得新鲜 providerBalanceMinor，在同一用户资金串行区重新计算 reservedMinor 与 availableMinor。
  4. 确认预留有效、未过期、金额和币种匹配且没有 provider balance deficit。
  5. 仅此时调用 createReservationDebit，请求必须携带 fundReservationId 与 fundReservationVersion。
阻断规则：不存在接收裸 ORDER 或 GIFT 的直接 debit API。缺少任一 reservation 字段返回 VALIDATION_ERROR；版本过期或生命周期冲突返回 RESERVATION_CONFLICT；调用供应商之前即失败。

所有 Bot、Dashboard、内部 Job 与人工结案资金入口都必须经过同一业务 API。任何绕过 FundReservation 的 ORDER/GIFT debit 都是 P0 阻断缺陷。

9. 过期、幂等与超时

9.1 过期

  • 原生模式的业务 expiresAt 默认早于供应商到期 60 秒；最终值以经证据批准的配置为准。
  • 本地回退由业务 API 的过期 Worker 原子标记 RELEASED 并递增 version。
  • 捕获前必须同时确认本地预留未过期；原生模式还必须确认 provider hold ACTIVE。
  • HOLD_EXPIRED 只应用一次，阻止 capture；不得补发 fallback debit。

9.2 幂等键

hold:order:{orderId}:v1
hold:gift:{giftRequestId}:v1
capture:hold:{fundReservationId}:v{fundReservationVersion}
release:hold:{fundReservationId}:v{fundReservationVersion}
debit:reservation:{fundReservationId}:v{fundReservationVersion}
refund:{originalTransactionId}:{resolutionId}:v1

同键同指纹返回原结果；同键不同指纹返回 IDEMPOTENCY_CONFLICT。所有写操作 0 次盲重试；超时标为 UNKNOWN，并按原 provider ref 或原幂等键调用 getHold/getTransaction。

10. 状态与 Webhook

10.1 Hold 状态

状态含义动作
UNKNOWN写结果无法确认。锁住意图并 getHold。
PENDING供应商已受理但未 ACTIVE。不推进派单或礼物任务到已预留终态。
ACTIVEhold 有效并可捕获或释放。保持对账直到终态。
PARTIALLY_CAPTURED已捕获部分，仍有余额。按决议捕获或释放剩余。
CAPTURED全部捕获。写消费事实；后续逆向用 refund。
RELEASED剩余金额已释放。不得再次捕获。
EXPIRED / FAILED不可捕获。业务异常或释放，不改走直接 debit。

10.2 事件与顺序

Hold 事件固定为 HOLD_CREATED、HOLD_UPDATED、HOLD_CAPTURED、HOLD_RELEASED、HOLD_EXPIRED；交易事件为 DEBIT_UPDATED、REFUND_UPDATED，交易状态保持 UNKNOWN/PENDING/SUCCEEDED/FAILED。

  1. 按原始 bytes 验签、校验时间窗，再解析和规范化。
  2. 以 (provider, eventId) 去重；重复事件返回 2xx，不重复应用。
  3. 优先使用 provider sequence；否则使用 occurredAt 与终态优先级。
  4. CAPTURED/RELEASED/EXPIRED 或 SUCCEEDED/FAILED 不被迟到旧状态降级；终态冲突保留首个可信终态并立即告警。
  5. 找不到本地引用时隔离事件，不创建无归属的 FundReservation 或财务记录。

11. 对账与余额缺口

15 秒→60 秒→5 分钟→15 分钟→30 分钟人工任务
  • 原生 hold 的 UNKNOWN/PENDING/ACTIVE/PARTIALLY_CAPTURED 使用 getHold 对账；fallback debit 与 refund 的 UNKNOWN/PENDING 使用 getTransaction。
  • 超过窗口仍无法确认时创建 AUTOMATION_FAILURE 客服任务，reason code 为 PAYMENT_REVIEW；不新增资金意图。
  • Webhook 是加速通道，不替代主动查询。每次查询和人工证据都只追加记录。

11.1 供应商站外消费

本地回退不能阻止用户在供应商其他入口消费。系统在周期刷新、新建预留前、fallback capture 前和资金异常后重新读取 provider balance，并计算：

deficitMinor = max(reservedMinor - providerBalanceMinor, 0)
条件必须动作禁止动作
deficitMinor > 0阻止新预留，保留现有记录，发出 provider_balance_deficit_alert，创建 AUTOMATION_FAILURE（PROVIDER_BALANCE_DEFICIT）。静默缩减预留、自动释放掩盖缺口、绕过预留直接扣款。
capture 因站外消费余额不足返回 INSUFFICIENT_FUNDS，保持可审计状态并升级 L2/L3；需要集成处置时通知 L4。把预留改成已捕获、生成新 key 或让客服直接改库。

12. 错误对齐

对外错误与统一 OpenAPI 对齐；供应商内部错误在离开业务 API 前映射。错误响应使用统一的 code/message/retryable/details 结构。

场景公开错误码规则
缺少 fundReservationId 或 versionVALIDATION_ERROR调用供应商前拒绝。
新预留时可用余额不足INSUFFICIENT_AVAILABLE_BALANCE返回 shortfallMinor，不创建 FundReservation。
预留版本过期或状态不允许RESERVATION_CONFLICT刷新对象，仍复用原业务 intent key。
同键不同指纹IDEMPOTENCY_CONFLICT不创建替代写请求。
站外消费导致已有预留捕获失败INSUFFICIENT_FUNDS不写成功消费，进入缺口/客服路径。
取消预览与订单/预留版本不再匹配CANCELLATION_PREVIEW_STALE重新预览，不释放、不退款。
供应商终态与本地自动化终态冲突AUTOMATION_STATE_CONFLICT保留首个可信终态并升级人工处置。
所选模式能力不可用SERVICE_NOT_AVAILABLE停止新预留，不在生命周期中切换模式。
供应商故障或限流PROVIDER_UNAVAILABLE有限读取重试；写入先查询。
供应商写超时PROVIDER_TIMEOUT结果 UNKNOWN，锁定原意图并对账。
Webhook 签名或重放失败SIGNATURE_INVALID / REPLAY_REJECTED解析与业务变更前拒绝。
供应商认证或 Schema 错误SERVICE_UNAVAILABLE / PROVIDER_UNAVAILABLE不向客户端暴露供应商原文。

13. 权限与安全

级别资金相关能力
L1_SUPPORT查看脱敏余额/预留状态、认领任务、联系核对、备注并提交审批；不得 capture、debit、refund 或改配置。
L2_SUPERVISOR在已批准金额阈值内处理礼物和退款，处理余额不足与缺口任务；不能绕过 FundReservation。
L3_OPERATIONS继承 L1–L2，处理更高金额、异常结案与缺口处置；达到 ¥5,000 时提交 L4。
L4_ADMIN_OWNER继承 L1–L3，可在 step-up 后直接执行高额操作，并管理供应商、模式、证据、密钥和 Kill Switch；仍不可硬删除记录。
  • API Key、Bearer/OAuth Token、mTLS 私钥、绑定码、签名、原始 Webhook 正文和完整 external user id 不进普通日志。
  • Bot 与浏览器永远不持有供应商密钥；出站网络只允许 API/Adapter 工作负载。
  • 停机开关停止新预留；查询、Webhook、release、对账与已批准 refund 保持可用。既有 capture 是否继续由版本化事故策略明确控制。

15. 接入与验收

供应商接入核对清单.csv 要求每个启用决定都有责任人、证据和上线影响。能力声明、Mock 成功或供应商口头确认都不能代替阻断证据。

15.1 阻断证据

  • 能力发现响应与正式文档一致，并完成原生 hold 或本地回退的明确模式批准。
  • 原生模式具有 create/capture/release/get、幂等、查询、状态映射和 TTL 全量沙箱证据。
