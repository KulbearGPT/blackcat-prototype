# P0 状态枚举与数据约束

## 1. 文档定位

本文件是 `schema.prisma` 的业务约束说明。数据库 Schema 定义实体、外键、唯一键和查询索引；本文件定义合法状态迁移、跨表不变量、事务边界，以及 Prisma Schema 无法直接表达但首批 PostgreSQL migration 必须加入的约束。

统一原则：Sapphire Bot 与 React Dashboard 只调用业务 API；只有 API 访问 PostgreSQL。第三方系统是用户、余额、扣款和退款的资金事实来源；本地只保存业务事实、外部交易镜像和不可变流水。

## 2. 通用数据规则

### 2.1 标识与时间

- 核心业务主键使用 UUID；面向用户的 `public_id` 仅用于展示，不作为外键。
- Discord Snowflake 使用字符串保存，禁止转成 JavaScript `number`。
- 所有时间以 `timestamptz` 保存为 UTC；展示时再按服务器配置时区转换。
- API 写操作使用 `row_version` 或目标版本做乐观并发控制。

### 2.2 金额与币种

- 所有金额使用 `BigInt` 的 minor units，例如 `CNY 12000` 表示人民币 120.00 元。
- `currency` 使用大写 ISO 4217 三字码；prototype 单币种，但每条财务记录仍保存币种快照。
- 价格、预留、捕获、释放、退款、消费、收益、返佣和冲正金额均不得为负数。`PlayerEarningAdjustment.amount_minor` 与 `CommissionAdjustment.amount_minor` 必须 `>= 0`，方向只能由 `REVERSAL_DEBIT`、`CORRECTION_DEBIT`、`CORRECTION_CREDIT` 类型表达，禁止用负数反转方向。
- 同一事务中的订单、外部交易、退款、消费、收益和返佣币种必须一致。
- 金额不得通过 JavaScript 浮点数计算。API 使用 `bigint` 或十进制定点库，并在 JSON 层传输十进制字符串。

余额响应字段固定为 `providerBalanceMinor`、`reservedMinor`、`availableMinor`、`currency`、`fetchedAt`，且：

```text
reservedMinor = Σ 每笔有效 FundReservation 的 (CREATED 金额 - CAPTURED 金额 - RELEASED/EXPIRED 金额)
availableMinor = providerBalanceMinor - reservedMinor
```

`providerBalanceMinor` 来自第三方实时查询；`reservedMinor` 和 `availableMinor` 都是读取时派生值，任何表都不得持久化这两个聚合字段。有效预留状态为 `PENDING`、`ACTIVE`、`DISPUTED`、`PARTIALLY_SETTLED`；派生结果小于零必须作为数据损坏报警，不能截断为零。

### 2.3 快照

订单从 `DRAFT` 提交到 `PENDING_DISPATCH` 时，必须一次性保存：

- 服务目录版本、游戏/服务/区服名称和代码；
- 计价单位、单位数量、客户单价、陪玩结算单价；
- 客户总价、预期陪玩收益、币种和需求标签快照。

礼物请求创建时必须保存礼物代码、名称、价格、币种和播报模板快照。目录改价或下架不得改变历史订单、礼物、收益或返佣。

推荐归因是严格保密数据。受益人、被推荐人关系、`PROMOTER_FIRST_PURCHASE`/`PLAYER_LIFETIME` 类型、规则版本、比例、固定额、返佣金额和归因状态不得进入通用 `AuditLog.before_snapshot/after_snapshot`、`ApprovalRequest.payload_snapshot`、`StaffTask.context_snapshot`、`OrderEvent.payload`、Outbox payload、错误对象、通知或普通导出。通用快照只保存归因/返佣记录 ID 和非敏感动作结果；敏感字段只保存在受限关系列并通过专用 L2/L3/L4 查询返回。

### 2.4 乐观并发与 `expectedVersion`

API 的 `version` 和 `expectedVersion` 统一映射数据库字段 `row_version`。P0 需要版本保护的资源为：`User`、`PlayerProfile`、`Order`、`StaffTask`、`GiftRequest`、`FundReservation`、`ReferralAttribution`、`Commission`、`PlayerEarning`、`ApprovalRequest` 和 `OutboxEvent`。

每次更新必须使用一条原子条件更新，不允许先查询版本再无条件写入：

```sql
UPDATE target_table
SET business_field = $new_value,
    row_version = row_version + 1,
    updated_at = now()
WHERE id = $id
  AND row_version = $expected_version
RETURNING row_version;
```

- 返回一行：更新成功，响应 `version` 使用递增后的 `row_version`。
- 返回零行：资源不存在或版本已过期；确认资源存在后返回 `409 VERSION_CONFLICT`，不得产生部分写入。
- `ApprovalRequest.target_version` 是被审批对象的快照版本；`ApprovalRequest.row_version` 是审批请求自身的并发版本，两者不能混用。
- Outbox 人工重试也必须携带 `expectedVersion`，成功后递增版本；工作进程内部领取任务使用行锁或 `SKIP LOCKED`，同样递增版本。
- 幂等键解决重复请求，`row_version` 解决不同请求并发覆盖；两项机制必须同时保留。

## 3. 状态机

### 3.1 订单 `OrderStatus`

| 当前状态 | 合法下一状态 | 说明 |
|---|---|---|
| `DRAFT` | `PENDING_DISPATCH`, `CANCELLED` | 资料完整、再次读取第三方余额并成功创建订单 `FundReservation` 后提交；此时不捕获。草稿取消无资金动作。 |
| `PENDING_DISPATCH` | `ACCEPTED`, `CANCELLED`, `EXCEPTION` | 5 分钟无人接单只结束派单轮次并保持预留；用户取消、订单预留 30 分钟到期、拒绝或失败时释放，不创建退款。 |
| `ACCEPTED` | `IN_SERVICE`, `EXCEPTION`, `CANCELLED` | 客户与陪玩分别确认 readiness；正常路径只有两者都确认后才能进入服务中。取消先生成影响预览。 |
| `IN_SERVICE` | `PENDING_CONFIRMATION`, `EXCEPTION`, `COMPLETED`, `CANCELLED` | 直接完成/取消仅允许有权限的人工结案。 |
| `PENDING_CONFIRMATION` | `COMPLETED`, `IN_SERVICE`, `EXCEPTION`, `CANCELLED` | 30 分钟超时只创建任务，不自动完成；成功完成必须先捕获订单预留。 |
| `EXCEPTION` | 进入异常前的合法状态、`COMPLETED`, `CANCELLED` | 争议期间预留保持；结案按决议捕获、释放或两者组合，并写 `OrderResolution`。 |
| `COMPLETED` | 无 | 捕获成功后的历史事实不回退；后续退款通过追加记录表达。 |
| `CANCELLED` | 无 | 历史事实不回退；纠错通过追加记录表达。 |

订单状态每次改变必须在同一事务中追加 `OrderEvent`，并使 `order_events(order_id, sequence)` 连续递增。订单事件不承担当前状态查询，`orders.status` 是当前状态，事件表是完整历史。

Readiness 规则：

- 接单事务写 `accepted_at` 和 `readiness_due_at`，并清空旧的 `customer_ready_at`、`player_ready_at`。
- 客户确认只写 `customer_ready_at` 和 `CUSTOMER_READY_CONFIRMED`；陪玩确认只写 `player_ready_at` 和 `PLAYER_READY_CONFIRMED`。两者都非空后，同一事务写 `service_started_at`、`SERVICE_STARTED` 并迁移为 `IN_SERVICE`。
- 到达 `readiness_due_at` 仍未双方就绪时写 `READINESS_TIMED_OUT` 并创建客服任务，不能由陪玩单方面开始计费。
- `SERVICE_STARTED_OVERRIDE` 只允许受权限控制的兼容迁移或人工结案路径，必须保存原因、审批/任务 ID 和真实操作者；普通用户/陪玩 API 不暴露该动作。

### 3.2 用户状态 `UserStatus`

- `ACTIVE`：允许正常登录、下单和送礼。
- `PAUSED`：可恢复的运营暂停；允许查看历史和联系客服，但禁止创建新订单或礼物。该值与 API 的 `PAUSED` 直接一对一持久化。
- `SUSPENDED`：因风险或违规暂停业务能力，只允许有权限的 L3/L4 恢复。
- `DISABLED`：保留为 L4 执行的系统级终止状态，用于账号关闭、人员离场或合规停用；普通用户状态 API 不接受该值，不能把日常暂停写成 `DISABLED`。

状态更新必须携带 `expectedVersion`，原子递增 `users.row_version`，并写入审计日志。`ACTIVE/PAUSED/SUSPENDED` 是产品与 API 的 canonical 可写状态；`DISABLED` 仅是内部系统状态。

### 3.3 陪玩准入与可接单性

`PlayerReviewStatus`：

- `PENDING_REVIEW`：待运营审核，不可接单。
- `ACTIVE`：准入有效，可继续参与候选判断。
- `PAUSED`：运营暂停，不可接单。
- `SUSPENDED`：违规或风险停用，不可接单。

`PlayerAvailability`：

- `AVAILABLE`：陪玩主动表示可接单。
- `BUSY`：陪玩主动忙碌或系统分配了活跃订单。
- `OFFLINE`：陪玩关闭接单。

进入派单候选必须同时满足：`review_status = ACTIVE`、`availability = AVAILABLE`、`discord_presence = ONLINE`、服务要求标签全部匹配、用户状态为 `ACTIVE`、`active_player_slot_id` 未被其他订单占用。

配置 `required_presence=ONLINE` 是唯一 P0 规则。`IDLE`、`DND`、`OFFLINE` 和 `UNKNOWN` 均不进入候选池；不得使用“非离线即可”的宽松判断。Presence 只是一项候选条件，不能自动改变陪玩的业务可用性。

### 3.4 派单 `DispatchStatus`

`PENDING → ACTIVE → ACCEPTED | TIMED_OUT | CANCELLED | FAILED`。

- 一个订单按 `round` 递增创建派单轮次，同一轮唯一。
- `ACCEPTED` 必须同时写入 `accepted_player_id`、订单 `player_id` 和 `active_player_slot_id`。
- 抢单使用条件更新或行锁，只有仍为 `PENDING_DISPATCH` 且 `player_id IS NULL` 的订单能成功。
- 未抢到的候选人记为 `LOST_RACE`；不能把失败点击当成系统错误。
- 5 分钟超时只结束当前 `DispatchAttempt`。订单仍为 `PENDING_DISPATCH`，原 `FundReservation` 保持有效；用户选择继续等待时只创建新派单轮次，选择取消时才全额释放。订单提交 30 分钟仍无人处理则自动取消并释放；未捕获前不得称为退款。

### 3.5 礼物 `GiftRequestStatus`

| 当前状态 | 合法下一状态 | 说明 |
|---|---|---|
| `PENDING_REVIEW` | `PENDING_APPROVAL`, `REJECTED`, `EXPIRED`, `WITHDRAWN`, `FAILED` | 创建请求时已经按礼物价格预留；客服只核对品类、目标和沟通结果。 |
| `PENDING_APPROVAL` | `APPROVED`, `REJECTED`, `EXPIRED`, `WITHDRAWN`, `FAILED` | 按金额路由到 L2/L3/L4，批准人不得依赖人工余额计算。 |
| `APPROVED` | `CAPTURED`, `FAILED` | 批准后取得 `user_id + currency` 资金锁，再次读取第三方余额并捕获已有预留。 |
| `CAPTURED` | `ANNOUNCED`, `REVERSED` | 捕获成功后写消费和 Outbox；播报失败不回退捕获。 |
| `ANNOUNCED` | `REVERSED` | 保存播报频道和消息 ID。 |
| `REJECTED` | 无 | 最终拒绝并释放全部未捕获预留。 |
| `EXPIRED` | 无 | 到期终态并释放全部未捕获预留。 |
| `WITHDRAWN` | 无 | 用户在批准前撤回并释放全部未捕获预留。 |
| `FAILED` | `PENDING_REVIEW`, `PENDING_APPROVAL`, `APPROVED`, `REJECTED` | 重试前按 `failure_code` 指示的失败阶段恢复；预留是否保持由事件事实决定，禁止补写成功。 |
| `REVERSED` | 无 | 退款后终态，保留原礼物事实。 |

礼物只允许关联处于 `ACCEPTED`、`IN_SERVICE`、`PENDING_CONFIRMATION`，或完成后 24 小时内的订单。`receiver_id` 必须等于订单当前 `player_id`，客户端提供的其他接收人一律拒绝。礼物确认按钮只有在 `availableMinor >= priceMinor` 时可执行；不足时返回差额和充值入口。

### 3.6 客服任务 `StaffTaskType` 与 `StaffTaskStatus`

`StaffTaskType` 直接采用 API canonical 值，API 与 Prisma 不设置别名或多对一映射：

| Canonical 类型 | 用途 |
|---|---|
| `ORDER_ASSIST` | 普通订单协助、用户主动求助。 |
| `CANCELLATION_ASSIST` | 已接单取消或取消规则需要人工介入。 |
| `GIFT_REVIEW` | 强制人工核对礼物品类、目标和真实意愿；余额与预留由 API 计算。 |
| `PLAYER_START_LATE` | 接单后超过启动宽限期仍未开始。 |
| `PLAYER_NO_SHOW` | 陪玩爽约核对。 |
| `CUSTOMER_NO_SHOW` | 用户爽约核对。 |
| `SERVICE_INTERRUPTED` | 服务提前中断或无法继续。 |
| `COMPLETION_REVIEW` | 用户未在期限内确认完成。 |
| `DISPUTE` | 服务结果、金额或完成状态争议。 |
| `AUTOMATION_FAILURE` | 扣款、退款、派单、频道或面板自动化失败。 |

细分原因写入 `staff_tasks.reason_code` 和 `context_snapshot`。例如原先作为类型使用的 `PAYMENT_FAILURE`、`REFUND_FAILURE`、`DISPATCH_TIMEOUT` 改为 `AUTOMATION_FAILURE` 下的原因码；`ORDER_DISPUTE` 改为 `DISPUTE`；`GENERAL_ASSISTANCE` 改为 `ORDER_ASSIST`。这些旧名称不得再作为 `StaffTaskType` 持久化。

`OPEN → CLAIMED → VERIFIED → PENDING_APPROVAL → APPROVED | REJECTED → RESOLVED`。

允许的旁路：

- `OPEN/CLAIMED/VERIFIED/PENDING_APPROVAL → CANCELLED`：关联业务已失效或用户在执行前撤回。
- `CLAIMED → OPEN`：客服主动释放或会话失效；必须写审计。
- 无需审批的非资金任务可从 `CLAIMED/VERIFIED` 直接到 `RESOLVED`。

认领使用条件更新，只有 `OPEN` 状态的第一位客服成功。客服任务状态不替代订单、礼物或退款状态。

### 3.7 审批 `ApprovalStatus`

`PENDING → APPROVED | REJECTED | EXPIRED | CANCELLED`，终态不可恢复。

- L1 只能核对、提交和升级，不得形成最终 `ApprovalDecision`。
- L2 可批准礼物 `<= 200000` minor units、退款 `<= 50000` minor units。
- 高等级自动继承全部低等级执行权限，级别序固定为 `L1 < L2 < L3 < L4`；权限解析使用 `actor_rank >= required_rank`，不能按枚举值完全相等判断。
- 超过 L2 限额且低于 `500000` minor units 由 L3 或 L4 直接处理；单笔 `>= 500000` minor units 最低要求 L4。
- 业务审批允许申请人与最终决定人为同一工作人员；只要当前有效级别达到 `required_level` 即可直接执行。首次 L3/L4 账号升级仍要求现有 L4 确认，因为升级生效前目标账号尚未获得高级权限。
- 决定前重新读取目标 `row_version`、`payload_hash`、余额和策略；任一变化使请求失效并重新申请。
- `ApprovalDecision` 一张申请最多一条，且是不可变记录；高额和敏感动作仍要求原因、近期 step-up 与完整审计。

金额阈值从版本化策略读取；以上数值是 prototype 固定初始配置，币种为 CNY。

### 3.8 外部扣款与退款

`ExternalTransactionStatus` 只描述原始订单/礼物扣款镜像及其退款聚合结果：

- `PENDING → SUCCEEDED | FAILED | UNKNOWN`。
- `UNKNOWN → SUCCEEDED | FAILED`，必须依靠第三方查询或已验签 Webhook 收敛。
- `SUCCEEDED → PARTIALLY_REFUNDED | REFUNDED`，由成功退款累计结果驱动。
- 不允许把超时直接标记为 `FAILED` 后重新生成新的扣款幂等键。

`RefundStatus` 只描述单次退款交易过程，值严格为 `UNKNOWN`、`PENDING`、`SUCCEEDED`、`FAILED`。`UNKNOWN/PENDING → SUCCEEDED | FAILED`，未知结果通过查询或 Webhook 收敛。退款记录本身永远不使用 `PARTIALLY_REFUNDED` 或 `REFUNDED`；退款成功后，由 API 根据累计成功退款额更新原扣款镜像为 `PARTIALLY_REFUNDED` 或 `REFUNDED`。

本地表只镜像外部交易，不计算或保存可用余额。退款总额不得超过原扣款减去既有成功退款后的净额。

### 3.9 资金预留 `FundReservationStatus`

| 当前状态 | 合法下一状态 | 对 `reservedMinor` 的贡献 |
|---|---|---|
| `PENDING` | `ACTIVE`, `FAILED`, `RELEASED`, `EXPIRED` | 原额减已捕获/释放额；创建意图成功后立即占用额度。 |
| `ACTIVE` | `DISPUTED`, `PARTIALLY_SETTLED`, `CAPTURED`, `RELEASED`, `EXPIRED`, `FAILED` | 原额减已捕获/释放额。 |
| `DISPUTED` | `ACTIVE`, `PARTIALLY_SETTLED`, `CAPTURED`, `RELEASED` | 保持原额减已捕获/释放额，不允许自动释放。 |
| `PARTIALLY_SETTLED` | `DISPUTED`, `CAPTURED`, `RELEASED` | 原额减已捕获/释放额。 |
| `CAPTURED` | 无 | `0`。后续退款不回退预留。 |
| `RELEASED` | 无 | `0`。 |
| `EXPIRED` | 无 | `0`，必须有等于剩余金额的 `EXPIRED` 事件。 |
| `FAILED` | 无 | `0`；只有从未形成有效 hold/local reservation 的失败才可进入。 |

`FundReservationEventType` 完整集合为 `CREATED`、`ACTIVATED`、`CAPTURED`、`RELEASED`、`DISPUTED`、`DISPUTE_RESOLVED`、`EXPIRED`、`FAILED`。事件表只追加，`(fund_reservation_id, sequence)` 连续且唯一；`CREATED/CAPTURED/RELEASED/EXPIRED` 金额必须为正，其他事件金额必须为零。任何时点累计捕获额与累计释放/过期额之和不得超过原始 `amount_minor`。

- 订单在确认提交时创建预留，完成时捕获；用户取消、待派单预留到期、拒绝或失败时释放；5 分钟派单轮次超时不释放；争议保持；部分结案在一个事务中追加捕获和释放事件，使剩余为零。
- 礼物在创建请求时创建预留，批准后捕获；拒绝、过期或撤回时释放。
- 优先使用 `PROVIDER_NATIVE_HOLD`；第三方不支持时使用 `LOCAL_RESERVATION_FALLBACK`，但订单、礼物及未来任何消费入口都必须经过统一 API 和同一资金锁。
- `fund_reservations.status` 是当前状态缓存，金额事实以 append-only 事件为准；状态与事件累计不一致时拒绝资金写入并报警。

### 3.10 推荐计划与归因

`ReferralProgramType` 只有：

- `PROMOTER_FIRST_PURCHASE`：一次性固定额或首笔符合条件净消费百分比；每个归因最多一笔原始 `Commission`，形成后归因进入 `FULFILLED`，永久不再产生第二笔。
- `PLAYER_LIFETIME`：按版本配置，对每个符合条件的净订单和/或净礼物消费各产生最多一笔原始 `Commission`，持续到归因被纠错或撤销。

`ReferralProgramVersionStatus`：`DRAFT → ACTIVE → RETIRED`，每种 `program_type` 同时最多一个 `ACTIVE` 版本。版本启用后规则字段不可覆盖；修改固定额、比例、订单/礼物资格或币种必须创建新版本。

`ReferralAttributionStatus`：

| 当前状态 | 合法下一状态 | 说明 |
|---|---|---|
| `ACTIVE` | `FULFILLED`, `SUPERSEDED`, `REVOKED` | 两个计划共用唯一 `active_attribution_key`，一个被推荐用户只有一个有效来源。 |
| `FULFILLED` | `SUPERSEDED`, `REVOKED` | 仅 `PROMOTER_FIRST_PURCHASE` 使用；仍占用唯一归因键，防止普通流程二次绑定。 |
| `SUPERSEDED` | 无 | L3/L4 纠错改绑后的历史记录。 |
| `REVOKED` | 无 | L3/L4 撤销后的历史记录。 |

归因来源类型只有 `INVITE_CODE`、`ADMIN_MANUAL`、`MIGRATED_LEGACY`。每条归因恰好一个 `source_type`；邀请码只保存不可逆 `source_reference_hash`，手工绑定不保存伪造的邀请码引用。改绑必须在串行化事务中先把旧归因置为 `SUPERSEDED` 并清空活动键，再插入带 `replaces_attribution_id` 的新归因，同时写不含敏感字段的审计记录。

一个被推荐用户不能同时属于两个计划。绑定事务必须在首笔符合条件消费之前检查 Discord 和第三方账号映射，拒绝同账号、自荐和直接互荐。普通用户 API、Discord、错误、通知和通用导出永远不得泄露受益人、计划、比例、金额或状态。

### 3.11 陪玩收益与返佣

`PlayerEarningStatus` 与 `CommissionStatus` 均为：

`PENDING → CONFIRMED → PAID`，发生全额冲正后可标记 `REVERSED`。

- 原始金额字段创建后不可修改。
- 退款或纠错分别追加 `PlayerEarningAdjustment`、`CommissionAdjustment`。这是唯一 canonical 冲正形态。
- 部分冲正时原记录保持原生命周期状态，净额通过原始金额减调整合计计算；只有累计冲正达到原始金额时标记 `REVERSED`。
- `PAID` 仅表示工作人员已在外部完成支付并做标记，不触发自动转账。
- 不创建第二条 `PlayerEarning` 或 `Commission` 作为冲正，也不在原记录增加 `reversalOf` 自关联。API 查询应返回原记录及其 `adjustments`，或返回独立 adjustment DTO；`CREATE_REVERSAL` 动作的数据库结果必须是新增 Adjustment。
- 原记录的状态确认、支付和全额冲正标记使用 `expectedVersion` 更新并递增 `row_version`；Adjustment 自身只追加，不需要可变版本。

### 3.12 Outbox、幂等与 Webhook

- Outbox：`PENDING → PROCESSING → COMPLETED | FAILED | CANCELLED`；可重试失败项通过增加尝试次数并重新进入 `PENDING`，不得创建相同 `dedupe_key` 的新任务。
- Idempotency：`IN_PROGRESS → COMPLETED | FAILED`；唯一作用域固定为 `client_id + operation + actor_key + key`。同一作用域且请求哈希相同返回首次结果，请求哈希不同返回冲突；`actor_key` 必须是 `USER:<uuid>`、`STAFF:<uuid>` 或 `SYSTEM:<stable-name>`，不得用 nullable 外键直接组成唯一键。
- Webhook：`RECEIVED → PROCESSING → PROCESSED | FAILED | IGNORED`；无效签名记为 `IGNORED` 并写安全审计，不进入业务事务。
- Discord Role 同步事件以 `(source, source_event_id)` 唯一；同一上游事件重放返回既有结果，不得再次提升、降级或增加 `permissions_version`。

## 4. 必须保持的跨表不变量

### 4.1 活跃订单槽位

活跃订单状态为：`DRAFT`、`PENDING_DISPATCH`、`ACCEPTED`、`IN_SERVICE`、`PENDING_CONFIRMATION`、`EXCEPTION`。

- 活跃订单的 `active_customer_slot_id` 必须等于 `customer_id`；终态必须为 `NULL`。
- 有陪玩的活跃订单，其 `active_player_slot_id` 必须等于 `player_id`；无陪玩或终态必须为 `NULL`。
- 两个槽位各自唯一，因此一个用户和一个陪玩分别最多只有一个活跃订单。
- 订单进入终态时，在同一事务中释放槽位；不得提前释放再异步完成结案。

### 4.2 目录版本

- 一个服务项目最多一个 `ACTIVE` 版本，`active_offering_key` 在启用版本中等于 `service_offering_id`，其他版本为 `NULL`。
- 一个礼物项目最多一个 `ACTIVE` 版本，`active_gift_key` 的规则相同。
- 双价格、计价单位和最低单位必须为正数；缺少客户价或陪玩结算价不得激活。
- 被订单或礼物引用的版本禁止更新价格和名称；改价创建新版本，旧版本改为 `RETIRED`。

### 4.3 交易来源

- `fund_reservations` 必须且只能关联一个来源：订单或礼物；`ORDER` 必须只关联订单，`GIFT` 必须只关联礼物。订单或礼物各最多一笔预留。
- `external_transactions` 必须且只能关联一个来源：订单或礼物；canonical 新写入还必须关联同来源、同用户、同币种的 `FundReservation`。
- `ORDER_CHARGE` 只能关联订单，且只能在订单完成/结案捕获时形成；`GIFT_CHARGE` 只能关联礼物，且只能在批准后捕获时形成。
- 订单或礼物最多一条外部扣款镜像；重试更新同一记录，不能新增扣款记录。
- 退款必须关联原始成功扣款，并且订单/礼物来源与原扣款一致。
- 消费原始借方必须关联成功外部扣款；退款贷方必须关联成功退款和原消费。

### 4.4 推荐归因与返佣

- `active_attribution_key` 在 `ACTIVE/FULFILLED` 时必须等于 `referred_user_id`，在 `SUPERSEDED/REVOKED` 时必须为 `NULL`；其唯一键在两个计划之间共用。
- 禁止 `beneficiary_user_id = referred_user_id`；`replaces_attribution_id` 必须指向同一被推荐用户的旧记录，且不能自引用。
- `active_program_key` 在 `ACTIVE` 版本中必须等于 `program_type`，其他状态必须为 `NULL`；每种计划最多一个活动版本。
- `FIXED_MINOR` 必须 `fixed_amount_minor > 0` 且 `rate_bps IS NULL`；`NET_SPEND_BPS` 必须 `fixed_amount_minor IS NULL` 且 `rate_bps BETWEEN 1 AND 10000`。`PLAYER_LIFETIME` 只能使用 `NET_SPEND_BPS`。
- 推荐归因必须在被推荐人的首笔符合条件消费之前绑定；检查和插入在同一串行化事务内完成。
- API 必须同时比较 Discord 和第三方账户映射，拒绝同账号、自荐和直接互荐。
- `commissions.source_consumption_entry_id` 全局唯一，保证每个符合条件消费来源最多一条原始返佣。
- PostgreSQL 部分唯一索引保证每个 `PROMOTER_FIRST_PURCHASE` 归因最多一条原始返佣；`PLAYER_LIFETIME` 则按每个符合条件的消费来源持续生成。
- `Commission.referral_attribution_id` 是唯一 canonical 归因外键；禁止保留或继续读取泛化的 `referral_id`。

### 4.5 权限映射

- Discord Role 只产生“请求级别”；API 使用 `staff_accounts.level/status/permissions_version` 解析真实权限。
- L1/L2 Role 可自动生效；首次 L3/L4 进入 `PENDING_ELEVATION`，由现有 L4 确认。
- L4 首次初始化只允许一次性 bootstrap；之后 L4 升级必须由另一名有效 L4 确认。
- 降级、停用或 Role 移除必须增加 `permissions_version` 并撤销全部旧会话。
- `active_mapping_key` 确保同一 Guild + Discord Role 只有一个当前映射；`active_level_key` 确保同一 Guild + Level 只有一个当前映射。

### 4.6 Guild Bot 配置

- `GuildBotConfig` 映射到 `guild_bot_configs`；`guild_id` 是主键，因此每个 Guild 恰有至多一条当前配置，`config_json` 只能保存通过业务配置 Schema 校验的批准字段。
- `version >= 1`。`updateBotConfig` 必须以 `guild_id + expectedVersion` 条件更新并原子递增；零行更新返回 `409 CONFIG_VERSION_CONFLICT`，禁止先读版本后无条件覆盖。
- 当前行更新与 `guild_bot_config_events` 插入必须在同一事务中完成。事件的 `version` 是更新后的版本，`(guild_id, version)` 唯一，并保存 `changes_json`、`previous_values_json`、原因、真实 staff actor、来源和时间。
- `GuildBotConfigEvent` 映射到 `guild_bot_config_events`，只允许 `INSERT`，禁止 `UPDATE/DELETE`；`GuildBotConfig` 当前行也不得硬删除。纠错必须提交新的版本和新事件。
- API 是最终授权点：`bot_config.read` 和 `bot_config.operational.manage` 最低 L3；`player_role_id`、`staff_l1_role_id`、`staff_l2_role_id`、`staff_l3_role_id`、`staff_l4_role_id` 还要求 L4 与 `bot_config.security.manage`。Discord Role 只用于身份同步，不能直接授权配置写入。
- Token、数据库连接、支付密钥、Webhook/OAuth Secret，以及服务、价格、礼物目录和返佣规则不得进入 `config_json` 或事件 JSON。
- Bot 启动/重启时从 `getBotConfig` 按 Guild 重建缓存；更新成功后刷新对应缓存。缓存不是授权、版本或审计事实来源。

## 5. PostgreSQL migration 必加约束

Prisma Schema 当前不能完整表达跨列 `CHECK`、条件关系和不可变触发器。首批 migration 必须加入等价约束；以下名称为推荐稳定标识。

```sql
ALTER TABLE service_catalog_versions
  ADD CONSTRAINT chk_service_prices_positive
  CHECK (
    billing_unit_minutes > 0 AND minimum_units > 0 AND
    customer_unit_price_minor > 0 AND player_unit_payout_minor > 0
  );

ALTER TABLE gift_catalog_versions
  ADD CONSTRAINT chk_gift_price_positive
  CHECK (price_minor > 0);

ALTER TABLE guild_bot_configs
  ADD CONSTRAINT chk_guild_bot_config_version_positive CHECK (version >= 1);

ALTER TABLE guild_bot_config_events
  ADD CONSTRAINT chk_guild_bot_config_event_version_positive CHECK (version >= 1);

ALTER TABLE orders
  ADD CONSTRAINT chk_order_readiness_times
  CHECK (
    (customer_ready_at IS NULL OR accepted_at IS NOT NULL) AND
    (player_ready_at IS NULL OR accepted_at IS NOT NULL) AND
    (service_started_at IS NULL OR accepted_at IS NOT NULL)
  );

ALTER TABLE gift_requests
  ADD CONSTRAINT chk_gift_price_positive CHECK (price_minor > 0),
  ADD CONSTRAINT chk_gift_terminal_times CHECK (
    (status <> 'CAPTURED' OR captured_at IS NOT NULL) AND
    (status <> 'WITHDRAWN' OR withdrawn_at IS NOT NULL)
  );

ALTER TABLE fund_reservations
  ADD CONSTRAINT chk_fund_reservation_source CHECK (
    (source_type = 'ORDER' AND order_id IS NOT NULL AND gift_request_id IS NULL) OR
    (source_type = 'GIFT' AND order_id IS NULL AND gift_request_id IS NOT NULL)
  ),
  ADD CONSTRAINT chk_fund_reservation_amount_positive CHECK (amount_minor > 0),
  ADD CONSTRAINT chk_fund_reservation_mode CHECK (
    (mode = 'PROVIDER_NATIVE_HOLD' AND provider IS NOT NULL) OR
    (mode = 'LOCAL_RESERVATION_FALLBACK' AND provider_hold_ref IS NULL)
  );

ALTER TABLE fund_reservation_events
  ADD CONSTRAINT chk_fund_reservation_event_amount CHECK (
    (event_type IN ('CREATED', 'CAPTURED', 'RELEASED', 'EXPIRED') AND amount_minor > 0) OR
    (event_type IN ('ACTIVATED', 'DISPUTED', 'DISPUTE_RESOLVED', 'FAILED') AND amount_minor = 0)
  ),
  ADD CONSTRAINT chk_fund_reservation_event_actor CHECK (
    num_nonnulls(actor_user_id, actor_staff_id) <= 1
  );

ALTER TABLE cancellation_previews
  ADD CONSTRAINT chk_cancellation_preview_requester CHECK (
    num_nonnulls(requested_by_user_id, requested_by_staff_id) = 1
  ),
  ADD CONSTRAINT chk_cancellation_preview_amounts CHECK (
    release_amount_minor >= 0 AND refund_amount_minor >= 0
  ),
  ADD CONSTRAINT chk_cancellation_preview_reservation_version CHECK (
    (fund_reservation_id IS NULL) = (reservation_version_snapshot IS NULL)
  ),
  ADD CONSTRAINT chk_cancellation_preview_status_times CHECK (
    (status <> 'APPLIED' OR applied_at IS NOT NULL) AND
    (status <> 'INVALIDATED' OR invalidated_at IS NOT NULL)
  );

ALTER TABLE referral_program_versions
  ADD CONSTRAINT chk_referral_program_active_key CHECK (
    (status = 'ACTIVE' AND active_program_key = program_type) OR
    (status <> 'ACTIVE' AND active_program_key IS NULL)
  ),
  ADD CONSTRAINT chk_referral_program_eligible_source CHECK (
    eligible_order_spend OR eligible_gift_spend
  ),
  ADD CONSTRAINT chk_referral_program_award_rule CHECK (
    (award_mode = 'FIXED_MINOR' AND fixed_amount_minor > 0 AND rate_bps IS NULL) OR
    (award_mode = 'NET_SPEND_BPS' AND fixed_amount_minor IS NULL AND rate_bps BETWEEN 1 AND 10000)
  ),
  ADD CONSTRAINT chk_player_lifetime_percentage CHECK (
    program_type <> 'PLAYER_LIFETIME' OR award_mode = 'NET_SPEND_BPS'
  );

ALTER TABLE referral_attributions
  ADD CONSTRAINT chk_referral_attribution_not_self CHECK (
    beneficiary_user_id <> referred_user_id
  ),
  ADD CONSTRAINT chk_referral_attribution_active_key CHECK (
    (status IN ('ACTIVE', 'FULFILLED') AND active_attribution_key = referred_user_id) OR
    (status IN ('SUPERSEDED', 'REVOKED') AND active_attribution_key IS NULL)
  ),
  ADD CONSTRAINT chk_referral_attribution_source CHECK (
    (source_type = 'INVITE_CODE' AND source_reference_hash IS NOT NULL) OR
    (source_type IN ('ADMIN_MANUAL', 'MIGRATED_LEGACY'))
  ),
  ADD CONSTRAINT chk_referral_attribution_replacement CHECK (
    replaces_attribution_id IS NULL OR replaces_attribution_id <> id
  );

ALTER TABLE commissions
  ADD CONSTRAINT chk_commission_amounts CHECK (
    base_amount_minor >= 0 AND amount_minor > 0
  ),
  ADD CONSTRAINT chk_commission_award_snapshot CHECK (
    (award_mode_snapshot = 'FIXED_MINOR' AND fixed_amount_minor > 0 AND rate_bps IS NULL) OR
    (award_mode_snapshot = 'NET_SPEND_BPS' AND fixed_amount_minor IS NULL AND rate_bps BETWEEN 1 AND 10000)
  );

ALTER TABLE player_earning_adjustments
  ADD CONSTRAINT chk_player_earning_adjustment_amount CHECK (amount_minor >= 0);

ALTER TABLE commission_adjustments
  ADD CONSTRAINT chk_commission_adjustment_amount CHECK (amount_minor >= 0);

ALTER TABLE external_transactions
  ADD CONSTRAINT chk_external_transaction_source
  CHECK (
    (type = 'ORDER_CHARGE' AND order_id IS NOT NULL AND gift_request_id IS NULL) OR
    (type = 'GIFT_CHARGE' AND order_id IS NULL AND gift_request_id IS NOT NULL)
  ),
  ADD CONSTRAINT chk_external_transaction_reservation
  CHECK (fund_reservation_id IS NOT NULL);

ALTER TABLE refunds
  ADD CONSTRAINT chk_refund_source CHECK (num_nonnulls(order_id, gift_request_id) = 1),
  ADD CONSTRAINT chk_refund_requester CHECK (num_nonnulls(requested_by_user_id, requested_by_staff_id) = 1),
  ADD CONSTRAINT chk_refund_amount_positive CHECK (amount_minor > 0);

ALTER TABLE order_resolutions
  ADD CONSTRAINT chk_resolution_target_terminal
  CHECK (target_status IN ('COMPLETED', 'CANCELLED')),
  ADD CONSTRAINT chk_resolution_amounts_nonnegative
  CHECK (refund_amount_minor >= 0 AND player_earning_minor >= 0);

ALTER TABLE approval_requests
  ADD CONSTRAINT chk_approval_amount_nonnegative
  CHECK (amount_minor IS NULL OR amount_minor >= 0);

ALTER TABLE idempotency_records
  ADD CONSTRAINT chk_idempotency_actor_key CHECK (
    (actor_user_id IS NOT NULL AND actor_staff_id IS NULL AND actor_key = 'USER:' || actor_user_id::text) OR
    (actor_user_id IS NULL AND actor_staff_id IS NOT NULL AND actor_key = 'STAFF:' || actor_staff_id::text) OR
    (actor_user_id IS NULL AND actor_staff_id IS NULL AND actor_key ~ '^SYSTEM:[A-Za-z0-9._:-]+$')
  );

CREATE UNIQUE INDEX uq_referral_one_effective_source
  ON referral_attributions(referred_user_id)
  WHERE status IN ('ACTIVE', 'FULFILLED');

CREATE UNIQUE INDEX uq_commission_promoter_once
  ON commissions(referral_attribution_id)
  WHERE program_type_snapshot = 'PROMOTER_FIRST_PURCHASE';

CREATE UNIQUE INDEX uq_commission_player_eligible_source
  ON commissions(referral_attribution_id, source_consumption_entry_id)
  WHERE program_type_snapshot = 'PLAYER_LIFETIME';
```

业务审批不设置“申请人与决定人必须不同”的数据库触发器。API 在事务内按审批快照的
`required_level`、当前操作者有效等级和权限码授权；达到最低等级的同一人员可以发起并决定。
决定前仍必须校验原因、近期 step-up、对象版本、预留与幂等键，并写入不可变审计。

员工 L3/L4 升权是独立例外：目标账号在新等级生效前无权确认自身升权。该限制由
`approveStaffRoleElevation` 在事务内比较目标账号与操作者，并通过唯一待确认升级记录防止并发绕过。

活跃槽位、目录激活键和角色映射键还需要触发器或 API 写入前校验，确保：

```text
active_customer_slot_id = customer_id       when order is active
active_customer_slot_id = NULL              when order is terminal
active_player_slot_id = player_id            when active order has a player
active_offering_key = service_offering_id    only for ACTIVE service version
active_gift_key = gift_catalog_item_id       only for ACTIVE gift version
active_mapping_key = guild_id + ':' + discord_role_id  only for current mapping
active_level_key = guild_id + ':' + target_level       only for current mapping
```

首批 migration 还必须实现以下跨表触发器；这些不是“仅由 API 约定”的可选规则：

1. `trg_fund_reservation_event_guard`：对目标 `fund_reservations` 行 `FOR UPDATE`；要求 `sequence = COALESCE(MAX(sequence), 0) + 1`。首事件必须是 `CREATED`、`amount_minor = fund_reservations.amount_minor`、`reservation_version = 1`；后续事件的 `from_status` 必须等于当前状态、`reservation_version = row_version + 1`。插入后累计 `CAPTURED + RELEASED + EXPIRED <= 原始 amount_minor`，并由触发器同步 `status/to_status`、`row_version`、`updated_at`。事件生命周期必须符合 3.9 表。
2. `trg_order_readiness_guard`：订单从 `ACCEPTED` 正常进入 `IN_SERVICE` 时要求 `customer_ready_at IS NOT NULL AND player_ready_at IS NOT NULL`，并要求同事务追加 `SERVICE_STARTED`。只有数据库会话已由受权用例设置事务局部标志 `SET LOCAL app.service_start_override = 'approved'` 且同事务追加 `SERVICE_STARTED_OVERRIDE` 时可绕过；连接池归还连接前不得保留该标志。
3. `trg_referral_attribution_guard`：`FULFILLED` 只能用于 `PROMOTER_FIRST_PURCHASE`；`replaces_attribution_id` 必须属于同一 `referred_user_id` 且旧记录为 `SUPERSEDED`。归因绑定/改绑时，对被推荐用户行加锁并确认尚无符合条件的消费。
4. `trg_commission_attribution_guard`：强制 `beneficiary_user_id`、`program_type_snapshot`、`program_version_snapshot`、`award_mode_snapshot`、`currency` 与归因及其计划版本一致；来源消费必须属于 `referred_user_id`、方向为 `DEBIT`，且条目类型符合版本的订单/礼物开关。
5. `trg_external_transaction_reservation_guard`：强制捕获镜像与预留的用户、币种、金额和订单/礼物来源一致；订单只允许在完成/结案用例捕获，礼物只允许在批准用例捕获。
6. `trg_guild_bot_config_event_immutable`：拒绝 `guild_bot_config_events` 的 `UPDATE/DELETE`；插入时锁定对应 `guild_bot_configs` 行，要求事件 `version = guild_bot_configs.version`，并验证事件前值、变更值与当前已校验 JSON 一致。API 数据库角色不得拥有该表的 `UPDATE/DELETE` 权限。

`staff_role_sync_events(source, source_event_id)`、`idempotency_records(client_id, operation, actor_key, key)`、`fund_reservation_events(idempotency_key)` 的唯一约束必须保留为数据库约束，不能只依赖缓存去重。

## 6. 不变流水与删除策略

### 6.1 禁止更新和删除

以下表只允许 `INSERT`，不得通过业务 API 更新或删除：

- `order_events`
- `fund_reservation_events`
- `order_resolutions`
- `staff_task_notes`
- `approval_decisions`
- `consumption_entries`
- `player_earning_adjustments`
- `commission_adjustments`
- `staff_role_sync_events`
- `guild_bot_config_events`
- `risk_events`
- `audit_logs`

生产 migration 应为这些表添加拒绝 `UPDATE/DELETE` 的触发器，并让 API 数据库角色不拥有 `DELETE` 权限。

### 6.2 可更新但禁止硬删除

订单、礼物、外部交易镜像、退款、客服任务、审批、陪玩收益、返佣、用户、人员账号和 Guild Bot 当前配置可以更新受控字段，但不能硬删除。纠错方式分别为：

- 订单：追加事件和结案记录；
- 交易、消费、收益和返佣：追加退款、贷方或调整记录；
- 用户、陪玩和工作人员：`SUSPENDED/DISABLED/PAUSED`；
- 服务和礼物目录：创建新版本并退役旧版本；
- Discord Role 映射和策略：创建新版本并结束旧版本。
- Guild Bot 配置：以 `expectedVersion` 创建新版本并追加不可变配置事件。

会话、过期幂等响应和已处理 Webhook 可依据运维保留策略清理，但清理不得级联删除任何业务或审计记录。

### 6.3 外键删除策略

- 业务事实、财务镜像和审计链统一使用 `RESTRICT`，防止删除上游实体造成历史断链。
- 仅非关键操作者引用允许 `SET NULL`，例如账号停用后保留“曾有操作者”事实；关键财务执行人仍使用 `RESTRICT`。
- 不在核心业务关系中使用级联删除。关联表也优先停用或版本化，避免误删历史匹配依据。

## 7. 原子事务边界

### 7.1 `user_id + currency` 资金锁

所有会改变预留、捕获、释放、退款或可消费额度的入口必须使用同一锁键；订单和礼物不能各有一套锁。精确顺序为：

1. 先以 `client_id + operation + actor_key + key` 创建/读取 `IdempotencyRecord`。
2. 在专用 PostgreSQL 连接上执行 `SELECT pg_advisory_lock(hashtextextended($user_id::text || ':' || upper($currency), 0));`。该 session lock 从最终第三方余额读取之前持有到本用例完成，跨 API 实例串行化同一用户同一币种；不同用户或币种可并行。
3. 每个短数据库事务先 `INSERT INTO user_currency_locks ... ON CONFLICT DO NOTHING`，再 `SELECT ... FROM user_currency_locks WHERE user_id = $1 AND currency = $2 FOR UPDATE`，随后从 `fund_reservation_events` 重算 `reservedMinor`。禁止读取某个缓存的余额字段。
4. 在 advisory lock 内再次读取第三方 `providerBalanceMinor`，验证 `availableMinor = providerBalanceMinor - reservedMinor`。外部 hold/capture/release 调用前后分别使用短事务登记意图和收敛结果；不得在网络等待期间保持普通数据库事务或行锁。
5. `finally` 在同一连接执行匹配的 `pg_advisory_unlock(...)`；连接异常关闭也必须释放 session lock。获取锁设置超时，超时返回可重试冲突，不得绕过锁执行。

### 7.2 用例事务

1. **订单提交并预留**：锁订单和 `user_currency_locks`，验证 `expectedVersion`、完整快照和可用余额，创建唯一 `FundReservation` 与 `CREATED/ACTIVATED` 事件，占用客户槽位，把订单改为 `PENDING_DISPATCH`，追加 `SUBMITTED`，写派单 Outbox。此时不得创建成功扣款/消费。
2. **礼物请求并预留**：锁订单、礼物和资金锁，验证目标与可用余额，创建礼物快照、唯一预留及事件、`GIFT_REVIEW` 任务。预留成功是请求创建成功的前置条件。
3. **并发接单**：锁订单，验证候选资格，占用陪玩槽位，写 `player_id/accepted_at/readiness_due_at`，清空 readiness，完成派单轮次，追加 `ACCEPTED`，写频道同步 Outbox。
4. **双方 readiness**：条件更新各自时间；第二个确认到达时在同一事务迁移 `IN_SERVICE` 并追加 `SERVICE_STARTED`。超时事务只追加 `READINESS_TIMED_OUT` 和创建客服任务。
5. **生成取消预览**：一致性读取订单、预留事件、成功捕获/退款和策略版本，写 `CancellationPreview(ISSUED)`。返回自动处理性、`releaseAmountMinor`、`refundAmountMinor`、处理方式和时效。
6. **执行取消/转客服**：锁订单、预览、资金锁和预留；要求预览未过期、状态为 `ISSUED`，且订单/预留版本与快照相同。重新计算影响完全一致后，原子追加释放/退款意图、订单事件、客服任务（如需）并把预览置为 `APPLIED`；不一致置为 `INVALIDATED` 并返回新预览。
7. **礼物批准并捕获**：锁礼物、审批、资金锁和预留；外部调用前再次读取余额。成功收敛事务追加 `CAPTURED` 预留事件，创建/更新唯一外部交易镜像和消费，更新礼物为 `CAPTURED`，按归因创建返佣，写播报 Outbox。播报失败只重试 Outbox。
8. **订单完成/人工结案并捕获**：锁订单、资金锁和预留；正常完成捕获订单金额，部分结案按决议捕获并释放剩余。成功收敛事务追加预留事件、外部交易与消费，创建收益/返佣，更新订单与槽位，追加完成/结案事件并写归档 Outbox。捕获未知或失败时订单不能进入 `COMPLETED`。
9. **退款成功收敛**：更新退款镜像和原交易累计状态、追加消费贷方、非负且方向在类型中的收益/返佣 Adjustment；不得覆盖原始金额。
10. **归因绑定/改绑与返佣**：按被推荐用户串行化，锁用户及现有归因；检查账号互斥与首笔消费，再写归因。生成返佣时锁归因并依赖数据库唯一索引，`PROMOTER_FIRST_PURCHASE` 最多一次，`PLAYER_LIFETIME` 每个合格消费来源最多一次。
11. **权限变更**：以 `(source, source_event_id)` 防重，创建角色同步事件、更新内部四级权限和 `permissions_version`、撤销旧会话、写审计；重复事件不产生第二次副作用。
12. **Guild Bot 配置**：锁定 `guild_bot_configs` 的 Guild 行，解析有效 staff actor 并逐字段执行 L3/L4 授权，校验 `expectedVersion`、Discord 对象和配置 Schema；同一事务递增版本、写入完整 `config_json`、追加 `guild_bot_config_events`。任一步失败均不改变当前配置，也不留下孤立事件。

第三方结果超时写 `UNKNOWN` 或保持预留 `PENDING/ACTIVE`，由固定幂等键查询或 Webhook 收敛；绝不换键重试或先释放额度。

## 8. 关键索引与查询目标

- 用户订单历史：`orders(customer_id, created_at)`。
- 陪玩订单历史：`orders(player_id, created_at)`。
- 活跃订单：唯一 `active_customer_slot_id`、`active_player_slot_id`。
- Readiness 超时：`orders(status, readiness_due_at)`。
- 派单超时：`dispatch_attempts(status, expires_at)`。
- 客服队列：`staff_tasks(status, type, created_at)`、`staff_tasks(claimed_by_staff_id, status)`。
- 礼物队列/到期：`gift_requests(status, created_at)`、`gift_requests(status, expires_at)`。
- 资金预留：`fund_reservations(user_id, currency, status)`、`fund_reservations(status, expires_at)`；来源订单/礼物分别唯一。
- 预留时间线：`fund_reservation_events(fund_reservation_id, sequence)` 唯一并按 `created_at` 查询。
- 取消预览：`cancellation_previews(order_id, created_at)`、`cancellation_previews(status, expires_at)`。
- 外部交易恢复：`external_transactions(status, created_at)`、`refunds(status, created_at)`。
- 消费历史：`consumption_entries(user_id, occurred_at)`。
- 推荐归因：按被推荐用户/受益人、状态和绑定时间检索；有效来源部分唯一索引；计划类型+版本唯一。
- 收益/返佣：按受益人、状态和时间的组合索引；推广者一次性奖励和陪玩逐来源奖励使用第 5 节两个部分唯一索引。
- 幂等：`idempotency_records(client_id, operation, actor_key, key)` 唯一。
- Role 同步：`staff_role_sync_events(source, source_event_id)` 唯一。
- Guild Bot 配置：`guild_bot_configs(guild_id)` 主键定位当前行；`guild_bot_config_events(guild_id, version)` 唯一并按 `actor_staff_id + created_at` 追溯。
- Outbox 消费：`outbox_events(status, available_at)`。
- 审计追溯：目标、操作者、请求 ID、Interaction ID 和结果索引。

## 9. Prototype 明确不建模的内容

- 预约时间/预约排期、排班、多人订单、指定陪玩、用户选择陪玩、陪玩试音/试听材料、订单加时和候补派单。
- 本地钱包、可用余额、银行卡、支付凭证或第三方密码。
- 多级返佣、自动提现、税务和自动工资发放。
- 复杂动态计价、优惠券、VIP、评价排名和跨 Discord Server 派单。

这些内容不得通过在现有 JSON 字段中偷偷扩展的方式进入 P0；新增范围必须先修改上位规格和数据模型版本。

## 10. 兼容与迁移顺序

1. **只加结构**：先创建 `user_currency_locks`、`fund_reservations`、`fund_reservation_events`、`cancellation_previews`、`referral_program_versions`、`referral_attributions` 及新 nullable 外键；此阶段旧写路径只读，禁止双写后各自决定余额。
2. **资金历史回填**：对每条既有订单/礼物成功 `external_transaction` 创建同用户、同币种、同金额的合成 `FundReservation(status = CAPTURED, idempotency_key = 'legacy:' || external_transaction.id)`，追加 `CREATED` 和 `CAPTURED` 事件并回填 `fund_reservation_id`。未扣款的活跃订单按现行业务事实创建 `ACTIVE` 预留；无法证明余额覆盖的记录进入人工迁移队列，不能静默激活。
3. **切换资金语义**：未来订单只在提交时 reserve、完成时 capture；礼物只在请求时 reserve、批准时 capture。既有“提交即扣款”订单保持合成已捕获事实，不倒放资金；取消走退款。确认无空外键后启用 `chk_external_transaction_reservation`。
4. **Gift 枚举**：旧 `PAID` 数据迁移为 `CAPTURED`，`paid_at` 重命名为 `captured_at`；新增 `PENDING_APPROVAL/EXPIRED/WITHDRAWN` 后再删除旧 enum value。应用发布必须先兼容读新旧值，再写新值，最后清理旧值。
5. **Readiness**：既有 `ACCEPTED` 行设置 `readiness_due_at`；既有 `IN_SERVICE/PENDING_CONFIRMATION/COMPLETED` 行用 `service_started_at` 回填两个 ready 时间，并追加 `SERVICE_STARTED_OVERRIDE` 迁移事件。迁移事件仅说明历史兼容，不能成为新订单正常路径。
6. **推荐迁移**：为每个旧规则创建版本。`CUSTOMER_TO_CUSTOMER → PROMOTER_FIRST_PURCHASE`，`PLAYER_TO_CUSTOMER → PLAYER_LIFETIME`；旧 `referrer_id/referred_id` 回填为 `beneficiary_user_id/referred_user_id`，来源标记 `MIGRATED_LEGACY`。先回填 `commissions.referral_attribution_id` 和计划快照，再删除泛化 `referral_id`/`referrals`；禁止长期双读双写。
7. **返佣冲突审计**：启用部分唯一索引前，按归因检查推广者多笔奖励、按消费来源检查重复陪玩奖励。重复记录不得任意删除；保留一条原始事实并以 `CommissionAdjustment` 冲正其余，记录受限审计。
8. **Adjustment 枚举**：旧 `REVERSAL → REVERSAL_DEBIT`。旧 `CORRECTION` 必须依据历史业务方向迁移到 `CORRECTION_DEBIT` 或 `CORRECTION_CREDIT`；无法确定方向时阻断迁移并人工确认。所有金额转绝对值后启用非负 `CHECK`。
9. **幂等与 Role 事件**：回填 `actor_key`；旧 `(client_id, key)` 冲突只有在 operation、actor 和 request hash 全部相同时可合并，否则阻断。Role 事件优先使用上游不可变事件 ID，历史行回填 `source_event_id = 'legacy:' || id`，再启用复合唯一键。
10. **派生余额清理**：删除任何遗留 `reservedMinor/reserved_minor/availableMinor/available_minor` 持久化列或缓存写路径。上线核对只比较第三方 `providerBalanceMinor`、事件派生 `reservedMinor` 和公式所得 `availableMinor`。
