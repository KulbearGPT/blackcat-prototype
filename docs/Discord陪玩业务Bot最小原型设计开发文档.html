1. 文档摘要

本文档是计划中 P0 开发的主规格，不表示产品已实现。Sapphire Bot 与 Dashboard 都是统一业务 API 的适配器客户端，不保存或重复实现业务规则；自有系统负责订单、资金预留、礼物、消费、返佣、陪玩收益、权限、审计和运营视图；第三方继续掌握用户账户事实、充值、支付、退款和真实余额。

核心原则：Discord 频道和 Dashboard 只承载交互。统一 API 负责业务状态、权限、幂等和并发；订单在提交时预留、完成时捕获，礼物在请求时预留、批准时捕获。资金摘要只使用 providerBalanceMinor、reservedMinor、availableMinor，其中 availableMinor = providerBalanceMinor - reservedMinor。

1.1 原型要证明什么

  • 普通用户无需学习大量斜杠命令，也能在 3–4 个主要步骤内完成下单。
  • 系统能创建仅相关人员可见的订单频道，并使用一条可持续更新的操作面板完成订单生命周期。
  • 符合标签、Discord 在线且设置为“可接单”的陪玩能在集中派单频道抢单，且第一位成功接单者唯一获得订单。
  • 用户和陪玩分别确认就绪，只有双方就绪才进入服务中；任何一方超时都转客服。
  • 礼物请求创建时就预留资金；客服只核对品类、目标和沟通结果，有权限的 L2–L4 批准时由 API 捕获既有预留，不人工计算余额。
  • PROMOTER_FIRST_PURCHASE 和 PLAYER_LIFETIME 两种互斥返佣计划可被准确结算、冲正和审计，且对被推荐用户严格保密。
  • 下单、派单、取消和完成默认自动，关键阶段都可被客服接管、暂停、转派或提交审批。
  • 统一运营工作台提供数字化派单、四级细粒度 RBAC/审计、自动运营记账轨迹和八项最小指标。

1.2 推荐技术取舍

业务形态

模块化单体。一个代码库，按用户、订单、派单、礼物、返佣拆模块。

运行进程

统一业务 API、Dashboard 前端与 Sapphire Discord Bot 分工运行；只有 API 访问 PostgreSQL，Bot 和 Dashboard 都通过 API 完成业务。

资金边界

优先使用供应商 hold/capture/release；不支持时由统一 API 维护本地预留，但不复制第三方真实余额。

2.2 启动前最优先六项

稳定 ID必须锁定的 P0 能力验收焦点
PL-01结构化需求与完整确认：游戏、服务、区服、时长、标签、备注、价格、可用余额和取消规则在提交前一次展示。用户不需跨页拼接信息；确认后仍由 API 重新校验。
PL-02原子资金预留：摘要统一显示第三方账户余额、预留金额和可用余额；availableMinor = providerBalanceMinor - reservedMinor。订单和礼物并发请求不能超支，reservedMinor 来自有效预留合计。
PL-03匹配透明与接单结果：显示匹配阶段、已通知候选数、超时下一步；接单后显示陪玩资料摘要和用户下一步。派单进度可见，且并发接单始终只有一名胜者。
PL-04陪玩工作台：独立显示资格、在线/可接单状态、当前订单、匹配订单、需求详情、倒计时、收益摘要和下一步动作。陪玩无需依赖管理员查询日常接单信息。
PL-05双方就绪再开始：用户和陪玩分别确认就绪，两者都就绪才自动进入服务中，超时转客服。不存在陪玩单方开始计费的正常路径。
PL-06取消影响预览：执行取消或申请客服前，显示可自动处理性、预计释放/退款金额、处理方式和时效。预览不构成承诺；执行结果仍由 API 按最新版本原子校验。

2.3 下一阶段与 Nice to Have

下一阶段统一编号为 P1-01 至 P1-06：陪玩档案展示与试音/审核材料、用户选择/指定陪玩、预约与排班、更完整的经营/对账/导出、多服务器或多渠道接入、移动端运营和通知中心。Nice to Have 包括额外登录、白标/自定义域名、多租户/复杂代理层级、大范围连接器、营销自动化、VIP、优惠券和活动系统。详细排序与边界统一见同交付包《下一阶段功能路线图》，本文不展开；P0 的 PL-01 至 PL-06 仍专指六个启动门禁。

2.4 明确非目标

  • 不自建充值、支付、退款、钱包或余额账本，不存储银行卡等支付敏感信息。
  • 本地 FundReservation 仅是业务预留事实，不是可手工编辑的钱包余额，也不取代第三方账户事实。
  • 不做多级返佣、公开通用推荐计划、自动提现、税务处理、代理层级或返佣资金发放。
  • P0 不做预约单、排班日历、陪玩试音/用户选择或指定陪玩、智能推荐算法、多人订单、动态竞价或跨服务器派单。
  • 不做优惠券、VIP、会员等级、独立客服工单产品、复杂客服 SLA、评价体系和完整财务报表。
  • 不保证在 Bot 离线期间继续提供 Discord 交互；后台数据与外部回调仍应可保存。
Prototype 限制：第一版只支持单个 Discord Server、单币种、每个订单一名陪玩、每名用户最多一个未结束订单频道。超出范围由管理员后台人工处理。

3. 功能流程（给非技术人员看的白话版）

这一节只解释“谁在什么时候做什么，以及最后会看到什么”。系统内部状态、数据库和 API 细节从下一节开始。

整体原则：能自动完成的事情默认自动完成；用户随时可以点“联系客服”；遇到余额、权限、退款或争议等异常时，系统自动把事情交给客服。送礼因为涉及对象、品类和公开播报，必须先由客服核对，并由有审批权限的主管或管理员批准。

3.1 用户第一次使用

用户在公共频道点击“创建订单”或“我的服务中心”。如果 Discord 账号还没有绑定业务账号，Bot 会请用户填写一次性绑定码。绑定成功后，用户可以看到第三方账户余额、预留金额、可用余额、进行中订单、消费历史和自己作为受益人的返佣/收益摘要，不需要学习斜杠命令。

3.2 下单

点击创建订单→进入私密频道→填写结构化需求→一次确认需求/价格/余额/取消→预留并派单

用户点击“创建订单”后，Bot 建立一个只有用户、Bot 和客服能看到的私密订单频道。首版只支持“尽快开始”的即时单；用户填写游戏、服务、区服、时长、标签和备注。提交前页面一次展示完整需求、价格、第三方账户余额、预留金额、可用余额和取消规则。确认提交时 API 原子创建订单预留，不立即扣款；无法自动处理时交给客服。

3.3 派单、接单和服务完成

订单提交后，Bot 只寻找已经审核为 ACTIVE、标签匹配、Discord 在线、标记为“可接单”且没有活跃订单的陪玩。用户可看到匹配阶段、已通知候选数和超时后的下一步。第一位成功接单者进入私密订单频道，用户看到陪玩资料摘要。用户和陪玩分别点击“我已就绪”，双方都就绪才自动进入服务中。陪玩申请完成后由用户确认；确认时 API 捕获订单预留，然后记录净消费、陪玩收益和符合条件的返佣。30 分钟未确认则转客服，不自动完成或捕获。

3.4 取消订单

用户在执行取消或申请客服前，先看到该状态能否自动处理、预计释放或退款金额、处理方式和时效。草稿无资金动作；已提交但未捕获的订单取消时释放预留。已开始、争议或需部分结案时，预留保持不变并转客服；结案决议原子捕获应消费部分、释放剩余部分，并明确陪玩收益。预览只是预计，最终由 API 按最新状态、版本和幂等键执行。

3.5 送礼物

选择礼物（目标由订单锁定）→预留资金并创建任务→L1+ 核对→有权限的 L2/L3/L4 批准→捕获与记账→公开播报
  1. 用户从已接单、服务中、待确认或完成后 24 小时内的订单面板选择礼物；接收陪玩由订单锁定，不允许用户另选。若价格超过可用余额，确认按钮禁用并显示差额与充值入口。
  2. 用户确认时，API 立即创建礼物预留和客服任务，防止并发礼物或订单超支；此时尚未捕获消费。
  3. 客服卡片显示礼物、目标、来源订单或“服务中心”来源、用户联系方式、语音频道链接、预留状态、有效期和可执行动作。
  4. L1 或更高等级客服核对品类、目标和沟通结果，然后提交审批；客服不人工计算余额，L1 不能直接捕获。
  5. 系统按金额把请求交给有权限的 L2/L3/L4。批准时 API 校验预留、版本和幂等键，捕获已有预留；本地预留模式下还会重新读取第三方余额并原子校验。
  6. 捕获成功后，自有系统保存礼物和消费事实，并通过 Outbox 播报。拒绝、过期或撤回则释放预留；播报失败只重试消息，不重复捕获。
语音链接规则：订单可以保存一个可选的 voice_channel_id。有链接时客服可一键加入；没有链接时处理卡片显示“用户当前未在订单语音频道”，客服改用私聊确认。prototype 不强制自动创建语音频道。

3.6 返佣计划与保密

PROMOTER_FIRST_PURCHASE 业务上称“老板带新”，通过邀请码或管理员手工绑定，按一次性固定额或首笔符合条件净消费比例结算，一次结算后永久结束。PLAYER_LIFETIME 表示陪玩带来的客人，按配置对长期符合条件的净订单和/或净礼物消费返点。一名被推荐用户只能有一个有效来源，两计划互斥，普通用户不可修改。只有受益人能在自己的“返佣/收益”中看到记录，来源用户默认脱敏；被推荐用户在用户 API、Discord、错误、导出和通知中永远不会得知受益人、关系类型、比例、金额或状态。

3.7 客服如何介入

必须人工确认

所有送礼请求。L1 可认领、联系用户和提交审批；最终批准/拒绝由 L2–L4 按金额和权限执行。

默认自动，可人工接管

下单、派单、取消和完成。用户主动求助、超时、规则命中或自动化失败时生成客服任务；客服可暂停自动化、转派或提交审批。

客服任务与订单状态分开保存。客服接管不会篡改历史记录；批准、拒绝、退款、转派等操作都会留下操作人、时间和原因。

3.8 客服和管理员如何使用 Dashboard

客服登录后台后先看到自己的待办和已认领任务。基础客服可以查订单、联系用户、写内部备注、核对送礼信息并提交审批，但看不到系统密钥，也不能直接扣款、退款、封禁用户或修改规则。客服主管处理常规审批和小额退款；运营负责人处理高金额、用户状态和业务配置；管理员/所有者只在人员权限、第三方集成或紧急停机等系统级操作时介入。

L1 核对并提交→系统判断权限和金额→L2/L3/L4 按规则审批→执行操作→写入审计并通知

3.9 Bot 和 Dashboard 为什么会得到相同结果

用户在 Discord 点击“取消订单”，或客服在 Dashboard 点击同一操作时，两个界面都会把请求交给同一套业务 API。API 根据操作者、订单状态、金额和权限决定自动取消、生成审批或拒绝操作，再把结果返回给界面。因此规则只维护一份，Bot 重启、Dashboard 改版或新增其他客户端都不会改变业务结果。

3.10 陪玩如何获得收益

陪玩通过独立工作台查看审核资格、在线/可接单状态、当前与匹配订单、需求详情、就绪/完成倒计时、收益摘要和下一步动作。每个上架服务同时配置客户单价和陪玩结算单价，订单提交时保存快照；订单捕获完成或人工结案后生成收益。运营可确认并人工标记已支付，退款时追加 Adjustment 冲正；原收益不允许覆盖或删除。

4.2 推荐实现栈

层建议原因
语言TypeScript / Node.jsSapphire 对 TypeScript 提供一等支持；Node.js 至少满足 Sapphire 官方要求的 18.x，生产使用当期仍受维护的 LTS。

5.4 高风险操作规则

  • 权限由 API 根据累积后的 permission_code、操作者级别、对象状态和金额共同判断；高等级自动拥有所有低等级执行权限，前端隐藏按钮不算权限控制。
  • Discord Role 映射只能授予不高于内部审批结果的有效权限；请求级别高于已批准级别时，API 始终取较低值。
  • Prototype 默认 L2 送礼审批上限为 ¥2,000、退款上限为 ¥500；超出后创建 approval_request 并升级给 L3。
  • 单笔送礼或退款达到 ¥5,000 时最低要求 L4；L4 可以直接发起并执行，不要求另一名人员复核。
  • 扣款、退款、封禁、规则变更、权限变更和集成变更必须填写原因、预览影响、二次确认并写入审计日志。
  • 财务流水、订单事件和审计日志只追加不覆盖；纠错只能创建冲正或更正记录。
  • 服务目录双价格、部分退款和陪玩收益确认/冲正属于财务相关操作，必须保存订单快照、原因和操作人。
  • L3/L4 执行敏感操作前必须完成近期二次验证；prototype 默认有效期 15 分钟。

  • @everyone：拒绝查看频道。
  • 下单用户：允许查看、发消息、读取历史、使用组件；不得管理频道。
  • 接单陪玩：接单成功后才加入；取消、转派或订单归档后移除。
  • L1–L4 工作人员：按任务或管理范围查看；L1 只能协助和提交审批，不能绕过 Dashboard/API 权限执行高风险动作。

  • 仅客服、管理员和 Bot 可查看；用户与陪玩不能进入。
  • L1 可以认领、打开订单频道、加入有权限的语音频道、私聊用户、写备注和提交审批。
  • 扣款、退款、拒绝、取消和关闭任务必须跳转 Dashboard 或调用同一业务 API，由 L2–L4 按权限和金额执行。

6. 核心用户流程

6.1 首次绑定与进入服务中心

  1. 用户点击公共入口的“我的服务中心”或“创建订单”。
  2. 系统检查 discord_user_id 是否已有有效绑定。
  3. 未绑定时弹出绑定 Modal，仅输入第三方系统签发的一次性绑定码。
  4. Payment Adapter 调用第三方用户验证接口；成功后保存映射，不保存第三方密码。
  5. Bot 返回仅本人可见的服务中心：第三方账户余额、预留金额、可用余额、进行中订单、最近消费以及当前用户作为受益人的返佣摘要。

6.2 创建订单

点击创建订单→账户校验→建立私密频道→填写结构化需求→一次完整确认→原子预留并派单
  1. 系统检查用户是否已有未结束订单；若有，提供“进入现有订单”按钮。
  2. 创建订单草稿和私密频道，发送并置顶一条常驻面板。
  3. 用户在订单常驻消息中通过 String Select 依次选择游戏、服务、区服、时长和标签；选择结果每次提交后由 Bot 更新同一条消息。补充备注通过单字段“补充备注 Modal”填写。P0 不采集未来开始时间，不展示试音材料，不让用户选择或指定陪玩。
  4. 业务服务根据目录中的计价单位、客户单价和最低单位数估价；订单同时保存目录版本、客户单价和陪玩结算单价快照。
  5. 确认页一次展示游戏、服务、区服、时长、标签、备注、价格、第三方账户余额、预留金额、可用余额和取消规则。
  6. 用户点击“确认并提交”；统一 API 再次读取账户事实，以业务来源、幂等键和版本原子创建 FundReservation，成功后进入待派单；此时不立即捕获消费。
  7. 用户点击“联系客服”、风控/规则命中或自动提交失败时，系统创建客服任务；否则不增加人工审批等待。

6.3 自动派单与抢单

  1. 根据订单的游戏、服务和区服标签筛选陪玩。
  2. 候选人必须同时满足：准入状态为 ACTIVE、标签匹配、Discord 在线、业务状态为“可接单”、无活跃订单。
  3. Bot 在集中派单频道发送订单卡片，仅 @ 候选陪玩或对应标签角色；同时向用户展示匹配阶段、已通知候选数和超时下一步。
  4. 陪玩点击“接单”；数据库事务以订单状态条件更新，第一位成功者获得订单。
  5. 系统将接单陪玩加入用户频道，向用户展示陪玩资料摘要和就绪下一步，更新双方卡片，并让原派单消息按钮失效。
  6. 指定时间无人接单时，将本轮 dispatch_attempt 标记为 DISPATCH_TIMEOUT。5 分钟超时保持订单预留和 PENDING_DISPATCH，用户可继续等待、取消并释放预留或联系客服；继续等待只创建新的派单轮次，不创建第二笔预留。订单提交 30 分钟仍无人处理时自动取消并释放，避免无期限占用。

6.4 服务与完成

  1. 接单后开始 10 分钟双方就绪宽限期；用户和陪玩在各自面板点击“我已就绪”，各自记录 readiness 事实。
  2. 只有 customer_ready_at 与 player_ready_at 都存在时，API 才自动将订单从 ACCEPTED 迁移到 IN_SERVICE 并记录实际开始时间。
  3. 任何一方未在宽限期内就绪都创建 PLAYER_START_LATE 客服任务，订单保持 ACCEPTED，不自动扣罚、捕获或释放。
  4. 陪玩完成后点击“申请完成”，等待用户确认。
  5. 用户点击“确认完成”后，API 先以预留的稳定幂等键捕获订单金额；捕获成功才完成订单，并生成消费事实、陪玩收益和符合计划的返佣。
  6. 用户 30 分钟未确认时创建 COMPLETION_REVIEW 任务；订单保持待确认，由客服决定完成、恢复服务或进入争议，不自动完成。
  7. 完成后频道进入只读状态，并在配置的保留期后归档。

6.5 取消订单

  • 草稿：用户可直接取消，不产生资金操作。
  • 待派单：用户可取消；API 释放订单预留，成功后关闭派单，不把未捕获金额描述为退款。
  • 已接单及以后：P0 不允许用户直接自动取消，系统先展示影响预览并创建 CANCELLATION_ASSIST 客服任务；争议期预留保持。
  • 用户主动点击“联系客服”、释放/捕获失败或状态变更时也创建客服任务；订单不得在资金结果未确认前显示为已取消或已完成。

人工结案必须提交 reason_code、应捕获金额、应释放金额、已捕获部分的退款金额、player_earning_minor 和处理说明；合计不得超过订单快照。资金与收益调整只追加记录；Adjustment 金额必须是非负绝对值，由 type 表达方向。

6.6 赠送礼物与播报

  1. 用户在符合条件的订单面板点击“赠送礼物”，选择启用的礼物；订单必填，接收陪玩由 order.player_id 推导并只读显示。
  2. API 返回三项资金摘要。若礼物价格超过 availableMinor，客户端禁用确认，显示差额和充值入口。
  3. 确认时 API 原子创建 gift_request、PENDING FundReservation 和 GIFT_REVIEW 客服任务。卡片显示礼物、目标、来源订单、用户联系方式、语音链接、预留状态、有效期和可执行动作。
  4. L1 或更高等级客服认领任务，通过语音或私聊核对品类、目标和沟通结果，然后将任务标记为“已核对，待审批”；L1 无权捕获或最终拒绝，任何客服都不人工计算余额。
  5. 系统按最低级别路由：L2 可批准 ≤ ¥2,000，L3 处理更高但低于 ¥5,000 的金额，≥ ¥5,000 由 L4 直接执行；任何更高级别也可执行低级别动作。
  6. 批准后 API 校验有效预留、版本和幂等键，捕获已有预留；本地预留模式在捕获前重新读取第三方余额。成功后创建消费事实；拒绝、过期或撤回后释放预留。
  7. Outbox 任务向配置的礼物频道播报，并更新播报状态；失败只重试播报，不重复扣款。
  8. 礼物是否计入返佣由配置决定，prototype 默认计入符合条件的净消费。

6.7 返佣计划

  • PROMOTER_FIRST_PURCHASE（老板带新）通过邀请码或管理员手工绑定，按固定额或首笔符合条件净消费百分比结算一次，结算后永久结束。
  • PLAYER_LIFETIME 适用于陪玩带来的客人，按配置对长期符合条件的净订单和/或净礼物消费返点。
  • 一名被推荐用户只能有一个有效来源，两计划互斥；绑定、改绑和冲正必须审计，普通用户不可修改。
  • 退款或结案追加 Adjustment 冲正，原返佣记录不覆盖或删除；Adjustment 金额为非负绝对值，由 type 表达方向。
  • 推广者/陪玩可查自己收益，来源用户默认脱敏。L1 默认不可看受益人和金额，L2 仅按任务有限查看，L3/L4 可绑定、纠错、确认和审计。
  • 被推荐用户的用户 API、Discord、错误、导出和通知不得暴露受益人、关系类型、比例、金额或状态。

6.8 客服接管

  1. 客服任务由用户主动求助、业务规则、派单/readiness/完成超时、用户/陪玩爽约、服务中断、资金或自动化错误以及强制礼物复核触发。
  2. 客服在集中处理频道点击“认领”，避免多人重复处理。
  3. L1 负责核对和提交；L2–L4 根据任务类型、金额与权限执行批准、拒绝、暂停/恢复自动化、转派、预留释放/捕获重试或关闭任务。
  4. 每个动作写入审计日志，并更新订单频道和客服卡片；客服任务状态不替代订单状态。

10. API 草案

统一前缀 /api/v1。Discord Bot 与 Dashboard 调用同一套业务接口和应用用例：Bot 使用受限服务账号并代表当前 Discord 操作者，Dashboard 使用 Discord OAuth2 登录后的后台会话。所有写接口接受 Idempotency-Key，响应包含 request_id；API 在每个入口执行相同的对象归属、权限码、金额、状态与审批策略。

复用原则：/admin 表示“需要工作人员权限”，不表示“只能由 Dashboard 调用”。Discord 客服卡片上的认领、核对、审批等按钮也可调用这些接口，但必须携带真实点击者上下文，并通过同一套 RBAC 和审计。

11.2 Sapphire 适配规则

  • 按钮、Select 和 Modal 统一由 InteractionHandler 路由；custom_id 只保存对象类型、ID 和动作，不塞业务数据。
  • 管理员命令先通过 Precondition 粗筛；按钮和 Modal 将真实点击者的 Discord User/Guild/Interaction ID 交给 API，随后由 API 解析内部身份并校验对象级权限。
  • Bot API Client 使用独立服务 Token；只有通过该服务身份的请求才能携带 Actor Header。Role ID 只允许提交到 Role Sync 用例，不作为普通业务接口的权限参数。
  • Listener 负责 Gateway 与框架事件；业务服务通过 Outbox/事件请求 Bot 更新消息，避免在 Listener 内写交易逻辑。
  • 部署入口和修复命令使用 Command；普通用户不需要面对大型斜杠菜单。
  • Sapphire container 只注册 API Client、logger、config 和 Discord renderer，禁止作为可随意写入的全局业务状态。

11.3 幂等与并发

  • Discord 的 interaction.id 作为交互幂等依据，重复提交返回首次结果。
  • 预留创建、捕获、释放和退款必须携带稳定 Idempotency-Key 与预期版本，重试不生成新键。
  • 接单采用条件更新：仅当订单仍为 PENDING_DISPATCH 且 player_id IS NULL 时成功。
  • 返佣使用来源唯一约束，避免事件重放产生重复记录。
  • 客服认领使用条件更新：仅当任务为 OPEN 时第一位客服可以认领。
  • 礼物批准使用 gift_request_id 派生固定捕获幂等键；重复点击不重复捕获。
  • 面板更新失败不回滚已经成功的业务事务；通过 Outbox 重试同步 Discord 展示。

创建频道失败Ephemeral 告知并提供重试。订单不进入可提交状态；后台记录失败原因。
两个陪玩同时接单胜者成功；其他人看到“订单已被接取”。数据库条件更新保证唯一获胜者。
派单无人接显示匹配阶段、已通知数和超时下一步。5 分钟超时保持原预留；继续等待只创建新派单轮次，取消或 30 分钟到期才释放。
客服任务被同时认领后点击者看到“该任务已由其他客服认领”。数据库条件更新保证唯一认领人。
礼物预留过期或版本变化“送礼预留已失效，请刷新后重新发起。”不捕获、不播报；原预留幂等释放，客服不手工改余额。
双方就绪超时双方看到“已转客服核对”。保持 ACCEPTED 和订单预留，创建 PLAYER_START_LATE；不单方开始或计费。
没有语音频道链接卡片显示“未检测到订单语音”。保留私聊用户和进入订单文字频道的入口，不阻断审核。
送礼请求长时间无人认领用户看到“等待客服确认”。10 分钟后提醒值班客服，不自动批准或拒绝。
Dashboard 会话失效“登录已过期，请重新登录。”返回 401，撤销前端缓存，不执行原操作。
Bot 服务身份或 Actor Context 无效“暂时无法验证操作身份，请稍后重试。”返回 401；不执行业务动作，记录 client_id、interaction_id 和拒绝原因。
Discord Role 尚未同步“工作人员权限正在同步，请稍后重试。”返回 STAFF_ACCOUNT_REQUIRED；不根据请求中自报 Role 临时放行。
首次获得 L3/L4 Role“高级权限等待管理员确认。”保持原级别并创建 ROLE_ELEVATION_PENDING；由现有 L4 审批。
Role 被移除或降级“权限已更新，请重新登录。”立即应用较低级别、递增 permissions_version、撤销 Dashboard 会话并拒绝旧权限请求。
后台越权操作“当前级别无权执行；需要 L2/L3/L4。”返回 403，不返回敏感对象内容，并记录被拒绝的权限码。
需要二次验证弹出重新验证流程。返回 428；验证成功后使用原 Idempotency-Key 重放一次。
审批已过期或数据已变化“原审批已失效，请刷新并重新提交。”不得沿用旧批准结果；创建新申请并关联旧记录。
礼物捕获成功、播报失败送礼成功，不提示重复付款。礼物保持 CAPTURED，Outbox 只重试消息，不重复捕获。
被推荐用户查询关系返回通用不可见结果。API/错误/日志/导出/通知都不暴露受益人、计划、比例、金额或状态。
退款失败“取消处理中，客服将跟进。”订单进入 EXCEPTION，不提前标记 CANCELLED。
服务价格已变化或下架“服务价格已更新，请重新确认。”草稿刷新目录版本；已提交订单继续使用原价格快照。
陪玩未通过审核或已有活跃订单“当前无法接取此订单。”拒绝接单，不泄露审核细节；刷新候选和派单卡片。
用户 30 分钟未确认完成双方看到“等待客服核对”。创建 COMPLETION_REVIEW；不自动完成、不生成陪玩收益。
陪玩/用户疑似爽约显示“客服正在核对订单”。创建 no-show 任务和风险事件；确认前不执行退款、补偿或扣罚。
人工结案金额不合法“退款或收益金额超出订单范围。”返回 422；不写结案、不调用第三方、不创建收益。
面板消息被删除管理员触发修复。根据订单数据库重建面板并更新 message_id。
用户/陪玩离开服务器无公开提示。保留历史；活跃订单提醒运营并进入异常处理。

12.1 日志与可观测性

  • 结构化日志至少包含 request_id、client_id、actor_source、interaction_id、order_id、user_id、staff_id、permission_code 和错误码。
  • 关键指标：Bot/API 调用失败率、第三方 API 延迟、待派单时长、开始服务超时数、完成确认超时数、爽约/中断任务数、待确认陪玩收益、未认领客服任务时长、审批等待时长、Role 同步失败/待确认数、403 次数和失败 Job 数。
  • 敏感凭证、绑定码和第三方原始签名不得写入日志；返佣受益人、关系、比例、金额和状态使用允许列表、权限分区和脱敏记录。

13. 客服与管理 Dashboard

Dashboard 是客服和管理人员的主要工作界面，不是只读报表。设计目标是让 L1 能快速处理日常任务，同时让财务、配置、权限和破坏性操作严格停留在对应级别。

13.0 当前 P0 竞品对齐能力

  • 统一订单与客服任务工作台，支持连续认领、详情抽屉和客服接管。
  • 数字化派单，包含候选筛选、已通知数、唯一接单、超时与人工转派。
  • 四级细粒度 RBAC、Discord Role 映射、金额审批、step-up 与不可删除审计。
  • 自动运营记账轨迹：余额快照、预留、消费、退款、陪玩收益、返佣和 Adjustment 进入同一业务时间线。

八项最小运营指标：今日订单、进行中订单、待处理任务、已完成净消费、礼物净消费、预留总额、派单成功率和异常数。所有指标必须定义时区、时间窗、币种、净额冲正口径与权限 scope。

13.3 L1 日常处理流程

  1. 从“我的任务”认领一项，详情抽屉展示必要的脱敏用户信息、订单时间线和 Discord 频道入口。
  2. 通过语音或私聊核对，将结论写入内部备注；内部备注不发送给用户。
  3. 若不涉及高风险操作，L1 可完成“已联系、资料已补充、升级异常”等非破坏性动作。
  4. 若需要退款、已接单取消、爽约/中断结案、收益调整、封禁或配置变更，L1 只能提交建议，填写原因码、建议退款、建议收益和证据说明。
  5. 任务进入“等待 L2/L3/L4”状态，L1 可查看进度但不能重复执行或修改审批内容。

13.4 高风险操作流程

选择动作→API 检查权限/金额/对象状态→预览影响并填写原因→二次验证或升级审批→幂等执行→审计与通知
  • Dashboard Dialog 必须显示用户、陪玩、订单、动作、客户退款、陪玩收益、资金去向和不可逆影响，不使用模糊的“确定吗”。Dashboard Dialog 是网页后台组件，不等同于 Discord Modal，也不受 Discord 的五组件和纵向布局限制。
  • 超权限或超金额时不显示伪成功，而是返回审批编号、所需级别和等待状态。
  • 审批人打开请求时重新读取对象和余额；数据变化后原审批失效，必须重新提交。
  • 重复点击、网络重试或多标签页执行同一操作时使用同一 Idempotency-Key，最多执行一次。
  • 完成后同时更新 Dashboard、Discord 订单面板和相关客服任务。

13.5 权限在界面中的表现

  • 无页面查看权限：导航隐藏，直接访问 URL 返回 403 页面。
  • 可查看但不可执行：显示禁用按钮和原因，例如“需要 L2”或“超过 ¥500 退款上限”，并提供“提交审批”。
  • 敏感字段按级别脱敏；第三方密钥、完整支付账号和原始签名不返回到前端。
  • 角色或权限被修改时立即撤销旧会话；下一次 API 调用不能继续沿用缓存权限。
  • 检测到 L3/L4 Discord Role 时显示“待确认级别”和来源 Role；确认前不展示高级操作。Role 被移除后不允许继续保留旧级别按钮。
  • 所有金额显示币种和最小单位换算结果；时间同时显示业务时区和相对等待时长。

14.3 /bot-config 基础配置面板

/bot-config 是 Guild 内 L3/L4 使用的管理命令，返回仅操作者可见的 ephemeral 配置消息。L3 可管理公共入口、订单分类、派单、陪玩工作台、礼物核对/播报、客服任务、运营告警、提醒时限和功能开关；L4 累积继承这些能力，并可管理陪玩及 L1–L4 Discord Role 映射。

  1. Command 从 Discord Interaction 解析 Guild 和操作者，调用统一 API 获取当前配置、版本与权限。
  2. 频道使用 Channel Select，身份组使用 Role Select，固定选项使用 String Select，数字和模板文本使用 Modal。
  3. 选择新值后先调用 validateBotConfigChange，显示旧值、新值和权限/类型检查；校验通过时 API 签发绑定操作者、Guild、版本、规范化变更和原因的短期 validationToken。
  4. 只有点击“确认应用”才携带 validationToken 调用 updateBotConfig；缺失、过期或内容不匹配时 API 拒绝，不能绕过预览直接保存。
  5. API 使用 expectedVersion 防止并发覆盖，保存前后值、原因、真实 staff actor 和时间；成功后立即刷新对应 Guild 缓存，使下一次派单、播报和功能开关判断使用新版本，Bot 重启后也从 API 重新加载。
程序边界：建议使用 commands/admin/bot-config.ts、interaction-handlers/bot-config/、presenters/bot-config-presenter.ts 和既有 Business API Client。Bot 不保存配置权限或业务规则；custom_id 只携带操作类型与短期会话 ID。Bot Token、支付密钥、数据库连接、游戏/价格、礼物目录和返佣规则不允许从 Discord 修改。

14.4 基础安全

  • 生产、测试使用不同 Discord App、数据库与第三方沙箱账号。
  • 数据库每日备份；至少验证一次恢复流程。
  • Dashboard 使用 Discord OAuth2 登录，并要求存在有效的本地 staff_accounts；L1/L2 可由 Role 同步创建，L3/L4 需确认。Discord Server 角色不能单独授予后台权限。
  • Bot 只能通过统一 API 访问业务数据；API 仅对有效 Bot 服务身份接受 Actor Header，并从数据库重新解析真实用户和权限。
  • 启动和 guildMemberUpdate 会对账 Role；L1/L2 可自动生效，首次 L3/L4 必须由现有 L4 确认，降级和撤权立即生效。
  • L3/L4 必须启用 MFA；敏感操作还需 15 分钟内的 step-up 验证。
  • 会话使用 HttpOnly、Secure、SameSite Cookie；禁止把访问令牌存入 localStorage；写操作校验 CSRF。
  • 角色修改、人员离职、异常登录和所有者操作可立即撤销全部会话。
  • 第三方 Webhook 必须验签并限制时间窗口，防止重放。
  • Bot Token、数据库密码和第三方 Key 不进入仓库或前端。

17. Prototype 验收标准

17.1 用户体验

  • 用户从公共入口开始，无需输入命令即可创建订单。
  • 已绑定用户从点击入口到提交有效订单，主要操作不超过 4 个阶段。
  • 订单频道仅用户、接单陪玩、Bot 和员工可见。
  • 订单面板始终显示最新状态、金额、需求和当前可用动作。
  • 个人余额、消费和返佣仅以私密/ephemeral 方式展示。

17.2 订单与派单

  • 字段不完整、余额不足或扣款失败时，订单不会进入待派单。
  • 只有双价格完整且已上架的服务可下单；订单保存目录版本、客户价和陪玩结算价快照。
  • Prototype 只接受即时单，不接受未来预约时间。
  • 候选陪玩必须 ACTIVE、标签匹配、Discord 在线、可接单且无活跃订单。
  • 并发接单只产生一名成功陪玩，数据库、订单频道和派单消息最终一致。
  • 接单成功后陪玩在 5 秒内被加入订单频道，原派单按钮失效。
  • 订单只能按定义的状态机迁移；越权或非法操作被拒绝并记录。
  • 待派单取消时，预留释放成功后才能进入已取消；未捕获资金不调用退款。
  • 用户求助、已接单取消和自动化异常都会生成可认领的客服任务。
  • 开始宽限、用户爽约和完成确认超时只创建客服任务，不自动退款、扣罚、完成或结算。
  • 人工结案必须原子地写入原因、退款、陪玩收益、订单事件和审计；任何一步失败都不产生半完成结果。

17.3 礼物、客服、消费与返佣

  • 每个送礼请求都会先进入客服处理频道，未经有权限的 L2–L4 按金额规则批准，不会调用第三方扣款。
  • 客服卡片显示用户、目标陪玩、礼物、金额、余额、订单以及语音链接或无语音提示。
  • 有权限的审批人批准时 API 校验并捕获请求创建时的原预留；每个礼物请求最多捕获一次并生成一条消费记录，客服不手工重算余额。
  • 礼物扣款成功后即使播报失败，也不会要求用户再次付款。
  • 订单和礼物消费均可在用户消费历史中分页查询。
  • 每个被推荐用户最多一个有效来源且两计划互斥；老板带新按配置的一次性固定额或首笔净消费比例结算，陪玩来源按配置对长期净消费返点。
  • 同一消费来源不会重复生成返佣；退款会产生可审计的冲正。
  • 礼物只能送给关联订单当前接单陪玩；客户端不能指定其他接收人。
  • 推荐关系拒绝自荐、重复账号及首笔符合条件消费后绑定。
  • 订单完成或人工结案生成陪玩收益；退款只通过冲正调整，原收益不可修改或删除。

17.4 Dashboard 与权限

  • L1–L4 登录后看到与角色匹配的导航、数据范围、按钮和金额阈值。
  • L1 可以完成查询、认领、联系、备注和提交审批，但无法直接扣款、退款、取消已接单订单、封禁、改配置或删除。
  • API 在任何入口都执行相同的权限码、scope、金额和对象状态检查；前端隐藏按钮不是唯一防线。
  • Bot 与 Dashboard 复用同一业务 API 和应用用例；同一操作者对同一对象执行同一动作时，状态迁移、金额规则、审批和审计结果一致。
  • Bot 业务请求具有可验证的服务身份和真实 Actor Context；客户端提交的 actor_id、level 或 Role ID 不能直接改变授权结果。
  • Discord Role 映射可在 Dashboard 查看；L1/L2 自动生效，首次 L3/L4 等待现有 L4 确认，Role 降级/移除立即撤权并撤销会话。
  • L2 超过 ¥2,000 送礼或 ¥500 退款时得到审批编号；不会执行半成功交易。
  • 达到 ¥5,000 的操作最低要求 L4；L4 可以直接执行，不强制独立申请人与批准人。
  • 角色变更立即撤销旧会话；L3/L4 敏感操作要求近期二次验证。
  • /bot-config 仅返回 ephemeral 配置消息；L3 可修改运营字段，L4 继承并可修改 Role 映射；所有修改先预览校验，再按版本确认应用并写不可删除审计。
  • 财务和审计数据只有冲正、禁用、归档或追加更正，没有任何硬删除路径。

17.5 运维与安全

  • 常见异常可从管理端查看并重试，不需要直接修改数据库。
  • 第三方回调验签、重复事件幂等、关键管理员操作有审计记录。
  • 每条 Bot/Dashboard 写操作审计均能追溯 client_id、actor_source、真实操作者和可选 interaction_id。
  • 日志不含 Bot Token、第三方 Key、绑定码或支付敏感数据。
  • 测试环境完成一次数据库备份恢复和 Bot 重启恢复演练。
  • 所有 P0 E2E 场景通过，且没有阻断目录计价、下单、扣款、接单、结案、收益或权限隔离的缺陷。
