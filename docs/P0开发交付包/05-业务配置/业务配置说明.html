P0 业务配置说明

把 P0 资金预留、双方就绪、取消预览、工作台、运营指标、双返佣计划及既有业务默认值转换为可校验、可导入、可版本化的配置基线，供统一业务 API、Sapphire Bot、React Dashboard 和测试共同使用。

上位规格：Discord 陪玩业务 Bot 最小原型设计开发文档 v0.7 配置版本：2.0.0 适用范围：P0 Prototype

1. 用途与文件

本目录不替代主规格。主规格决定业务边界；本目录负责把可配置内容变成开发团队可以直接读取和校验的格式。

business-config.example.yaml

完整开发示例。部署级开关为关闭，业务目录示例均未启用。

business-config.schema.json

JSON Schema Draft 2020-12，用于 API 启动和配置发布前校验。

seed-data.csv

可导入的扁平初始化清单，适合数据库种子和运营确认。

上位设计开发文档

唯一上位规格；状态机、数据模型、API 和验收均以该文档为准。

安全默认：示例配置设置为 deployment.enabled=false 和 example_only=true。未确认的游戏、区服、服务、双价格、礼物、陪玩标签及 Discord ID 均标记为 UNAPPROVED_EXAMPLE，不得进入生产。

2. 配置原则

原则执行方式
统一业务入口配置由统一业务 API 读取和执行业务校验。Bot 与 Dashboard 不复制价格、阈值、返佣或权限逻辑。
金额统一金额全部使用整数最小货币单位。人民币 ¥2,000 写作 200000，禁止浮点金额。
比例统一比例使用 basis points。2% 写作 200，5% 写作 500。
稳定标识业务 ID 使用小写字母、数字和下划线；原因码和状态使用大写稳定标识；Discord ID 保存为字符串。
版本化目录、双价格、礼物、返佣、权限和阈值变更创建新版本并记录操作者、原因和生效时间，不覆盖订单快照。
禁止硬删除订单、资金镜像、消费、收益、礼物、返佣、退款和审计记录没有硬删除配置；纠错使用冲正、禁用或追加更正。
资金单一入口所有订单、礼物及其他消费都必须经过统一业务 API；all_spend_via_unified_api 固定为 true，本地预留模式不能被其他入口绕过。
安全失败缺字段、引用不存在、配置未批准、Discord 权限不足或生产仍使用示例 ID 时，API 启动失败或拒绝发布。

3. 启动校验

API 是唯一读取业务配置并访问 PostgreSQL 的进程。推荐在 API 启动时先把 YAML 解析为对象，再用 JSON Schema 校验；Bot 与 Dashboard 从 API 获取公开配置和 capabilities。

3.1 Schema 强制校验

  • 金额为不小于零且不超过 JavaScript 安全整数范围的整数；正价格必须大于零。
  • PROMOTER_FIRST_PURCHASE 与 PLAYER_LIFETIME 两个计划必须同时存在，且比例使用 1–10000 basis points 的正整数。
  • Discord Snowflake 为 17–20 位非零开头数字字符串。
  • 稳定业务 ID 匹配 ^[a-z][a-z0-9_]{2,63}$；原因码使用大写格式。
  • 未批准示例必须同时满足 enabled=false，中文名称必须包含“示例”和“未确认”。
  • permissionCode 的 79 个稳定标识必须与角色配置、CSV 权限行及 API x-required-permissions 精确相等；不接受别名或迁移层。
  • access_control.inheritance.mode 固定为 CUMULATIVE：高级别自动继承全部低级别执行权限，级别顺序只能是 L1、L2、L3、L4。
  • L1 权限集合不能出现扣款、退款执行、人工结案、转派、目录修改、收益修改、权限修改等破坏性操作。
  • funding、readiness、cancellation、player_workspace 和 operations_dashboard 均为必填；运营指标固定为八项。
  • all_spend_via_unified_api 必须为 true；订单和礼物资金生命周期触发器不得被配置改写为其他阶段。
  • L2 送礼上限固定为 200000 分，退款上限固定为 50000 分；500000 分起最低要求 L4 直接执行，不强制独立复核人。

3.2 API 语义校验

  • 所有 ID 唯一；服务引用的游戏、区服和标签必须存在，启用服务只能引用已启用条目。
  • 每个启用服务必须同时具有客户单价和陪玩结算单价，并保存正整数版本号。
  • 礼物播报模板必须包含 {sender_name}、{receiver_name} 和 {gift_name}。
  • 同一 Guild 内频道 ID 和角色 ID 不得重复；Discord API 必须确认对象存在且 Bot 具备所需权限。
  • 只读时间不得晚于归档时间；归档 Job 只能移动频道和收紧权限，不能删除订单数据。
  • reservedMinor 必须由有效 FundReservation 剩余金额汇总得到，不能作为可编辑余额字段；availableMinor = providerBalanceMinor - reservedMinor。
  • 供应商原生 hold 可用时优先使用；使用 LOCAL_RESERVATION_FALLBACK 时，提交或批准前必须重读第三方余额并由统一 API 原子校验。
  • 推荐来源对被推荐用户在用户 API、Discord、错误、导出和通知中均不可见；L1 不能看到受益人或金额。
  • 生产环境要求 deployment.enabled=true、example_only=false，且不存在已启用的 UNAPPROVED_EXAMPLE。
  • 生产启动前必须关闭一次性 L4 bootstrap；已存在 L4 后再次设置 bootstrap 应拒绝启动。

字段迁移：旧键 order_category_id、archive_category_id、staff_channel_id 已从现行配置中删除，分别迁移为 private_order_category_id、order_archive_category_id、staff_task_channel_id。新配置、Schema、CSV 和通知引用只接受新键；部署迁移时应在同一版本变更中改名，禁止继续读取旧键或保留双写别名。

4.2 /bot-config 精简配置面

Slash Command 只开放常用运营配置，不把整份业务配置暴露到 Discord。时间与规则字段按下表校验，礼物播报模板只允许 {sender_name}、{receiver_name} 和 {gift_name} 三个占位符。

dispatch_max_rounds1–5 轮L3

completion_confirmation_minutes5–120 分钟L3
gift_review_reminder_minutes1–60 分钟L3

功能开关为 new_orders_enabled、auto_dispatch_enabled、gift_requests_enabled 和 maintenance_notice。关闭开关只阻止新的请求，不取消正在处理的订单、派单或礼物。

  • bot_config.read 与 bot_config.operational.manage 从 L3 起累积授予，L4 自动拥有；频道、通知 Role、时间、模板和开关属于日常运营配置。
  • bot_config.security.manage 仅授予 L4，用于陪玩及 L1–L4 身份 Role 映射；自有 API 必须结合有效 staff_account、等级和权限码最终授权，不能只信任 Discord Role。
  • 频道使用 Channel Select，Role 使用 Role Select；保存前检查对象属于当前 Guild、类型适用且 Bot 拥有查看、发送消息或管理频道/Role 的必要权限。
  • 命令消息仅操作者可见。每次修改先校验并展示旧值、新值和错误；API 校验通过后签发绑定操作者、Guild、版本、规范化变更和原因的短期 Token，“确认应用”必须携带该 Token，确认后记录原因、操作者、前后值和时间并立即刷新对应 Guild 缓存。
  • Bot Token、数据库连接、支付密钥、Webhook/OAuth Secret、游戏与服务目录、价格、礼物目录和返佣规则不在 Discord 中配置。

6. 资金、礼物与返佣

6.1 余额与预留

第三方系统继续提供账户事实和真实余额，自有系统记录 FundReservation。统一余额摘要只使用 providerBalanceMinor、reservedMinor、availableMinor、currency 和 fetchedAt；其中 reservedMinor 是有效预留剩余金额之和，availableMinor = providerBalanceMinor - reservedMinor。

配置P0 规则
预留模式PROVIDER_NATIVE_HOLD 或 LOCAL_RESERVATION_FALLBACK。优先供应商原生 hold/capture/release；供应商不支持时允许本地回退。
统一入口all_spend_via_unified_api=true 为不可修改约束。本地模式下，任何消费入口都不能绕过统一业务 API。
订单 TTL未接单订单预留默认 30 分钟；到期释放并创建客服任务。进入后续业务状态后由生命周期事件处理，不把预留当作可编辑倒计时余额。
礼物 TTL待批准礼物预留默认 30 分钟；拒绝、过期或撤回时释放。
余额过期第三方余额快照超过 30 秒即阻止新预留，主动刷新并告警。
预留缺口当 providerBalanceMinor < reservedMinor 时，阻止新消费、保留已有预留并创建客服任务；不得静默删减预留。
固定资金生命周期：订单在用户确认提交时预留，在订单完成时捕获；用户取消、待派单预留 30 分钟到期、拒绝或失败时释放，5 分钟派单轮次超时保持，争议时保持，部分结案按决议捕获并释放剩余。礼物在创建请求时预留，在批准时捕获，拒绝、过期或撤回时释放。每次创建、捕获和释放都必须携带业务来源、幂等键和版本，并由统一 API 做并发控制。

6.2 礼物目录与客服任务

  • 接收人固定为关联订单当前接单陪玩；客户端不能提交其他 receiver_id。
  • 礼物可在 ACCEPTED、IN_SERVICE、PENDING_CONFIRMATION 和完成后 24 小时内发起。
  • 所有礼物强制进入客服核对；L1 可认领和提交审批，但不能批准或扣款。
  • 用户确认请求时立即创建预留；礼物价格超过 availableMinor 时禁用确认，并显示差额与充值入口。
  • 批准时捕获已有预留，不由客服人工计算余额；本地模式再次读取第三方余额并原子校验。捕获成功后才写消费事实并经 Outbox 播报，播报失败只重试消息。
  • 示例中的三种礼物及价格均未获得业务批准，因此默认禁用。

6.3 两个返佣计划

计划受益人基数与结算
PROMOTER_FIRST_PURCHASE
业务称“老板带新”		PROMOTER首笔符合条件净订单或净礼物消费；奖励可配置为固定最小货币单位或百分比，只结算一次，结算后永久结束。
PLAYER_LIFETIMEPLAYER按配置选择净订单消费、净礼物消费或两者；对每笔长期符合条件的净消费按配置比例结算。
  • 邀请码和管理员手工绑定均可用；一个被推荐用户只允许一个有效来源，两个计划互斥，禁止自荐和重复身份互荐。
  • 被推荐用户不能通过用户 API、Discord、错误、导出或通知得知受益人、关系类型、比例、金额或状态，也不能自行绑定、改绑或纠错。
  • 推广者或陪玩只查看自己的收益，来源用户默认脱敏；L1 不看受益人和金额，L2 仅按任务有限查看，L3/L4 可执行经审计的绑定、纠错、确认和审计。
  • 退款与结案以追加 Adjustment 冲正；金额存非负绝对值，由 type 表达方向，原返佣和绑定记录不可覆盖或删除。
  • Prototype 仅记录、确认和人工标记发放，不自动转账。

7. 匹配、就绪、取消与工作台

7.1 陪玩标签

标签分为 GAME、SERVICE、REGION 和可选的 LANGUAGE。P0 自动派单强制匹配前三类；语言标签可以展示或人工筛选，但默认不作为阻断条件。示例标签均未确认并保持禁用。

7.2 派单候选与匹配透明

候选陪玩必须同时满足：准入状态 ACTIVE、业务状态 AVAILABLE、Discord Presence 为 ONLINE、无活跃订单，并匹配游戏、服务和区服标签。第一位事务性接单成功者胜出。用户侧按顺序展示 VALIDATING、MATCHING、NOTIFYING、AWAITING_ACCEPTANCE、ACCEPTED 或 TIMED_OUT，同时展示已通知候选数和超时下一步；接单后展示陪玩资料摘要与用户下一步。

7.3 双方就绪与取消预览

流程规则到期或异常
双方就绪订单从 ACCEPTED 出发，用户与陪玩分别确认；只有两者都就绪才自动进入 IN_SERVICE。10 分钟未完成双方就绪时暂停自动化并创建客服任务；陪玩不能单方面开始计费。
取消影响预览自动取消或申请客服前，展示可自动处理性、预计释放金额、预计退款金额、处理方式和时效。预览 60 秒过期；过期必须刷新。执行时 API 仍须原子重验，不把预览当作最终裁决。

系统不配置 order.start 权限，也不暴露绕过双方就绪的开始接口。需要人工判断时创建接管任务；客服只能处理阻塞原因，服务开始仍须满足双方就绪，旧调用一律拒绝并审计。

7.4 陪玩工作台

陪玩使用独立工作台查看资格、AVAILABLE/BUSY 状态、当前订单、匹配订单、需求详情、倒计时、收益摘要和下一步动作，并可设置可接单状态、事务性接单、确认就绪和请求完成。所有写操作仍调用统一业务 API。

7.5 运营工作台与八项指标

运营 Dashboard 支持接管、暂停、恢复、转派和提交审批。L1 只能提交审批；接管、暂停、恢复和转派最低为 L2，且 API 继续按对象状态、金额和动态审批级别校验。

指标 ID单位 / 窗口定义
TODAY_ORDER_COUNTCOUNT / BUSINESS_DAY业务时区当日创建的订单数量。
IN_PROGRESS_ORDER_COUNTCOUNT / POINT_IN_TIME当前处于派单、已接单、服务中或待完成确认状态的订单数量。
PENDING_STAFF_TASK_COUNTCOUNT / POINT_IN_TIME当前待认领或处理中且尚未解决的客服任务数量。
COMPLETED_ORDER_NET_CONSUMPTION_MINORMINOR / BUSINESS_DAY当日完成订单捕获金额减退款与追加冲正。
GIFT_NET_CONSUMPTION_MINORMINOR / BUSINESS_DAY当日礼物捕获金额减追加冲正。
ACTIVE_RESERVED_MINORMINOR / POINT_IN_TIME全部有效 FundReservation 剩余金额之和。
DISPATCH_SUCCESS_RATE_BPSBASIS_POINTS / BUSINESS_DAY当日成功接单数除以发起派单数并乘以 10000；分母为零时为零。
EXCEPTION_COUNTCOUNT / POINT_IN_TIME未关闭的余额过期、预留缺口、就绪超时、取消或自动化异常数量。

10. 导入与变更

10.1 推荐导入顺序

  1. 部署元数据、业务币种与时区。
  2. Discord Guild、频道和角色映射。
  3. 游戏、区服、服务、双价格和陪玩标签。
  4. 资金预留模式、余额过期/缺口策略及订单和礼物生命周期。
  5. 礼物目录、推荐绑定/隐私规则及 PROMOTER_FIRST_PURCHASE、PLAYER_LIFETIME 两个计划。
  6. 派单透明度、双方就绪、取消预览、陪玩工作台、运营控制、八项指标、其他超时、归档和通知策略。
  7. 79 个权限码、四级角色矩阵、金额门槛、MFA 和原因码。

10.2 CSV 约定

record_type + record_id 作为导入自然键；空列表示该类型不使用该字段。metadata_json 存放无法直接展开的一对多字段。导入器必须先做整批校验，再在单个数据库事务中写入；任何一行失败都不产生部分配置。

10.3 变更控制

  • L3 可管理服务、礼物、返佣、原因码、通知和普通业务阈值；每次发布必须填写 BUSINESS_CONFIG_CHANGE 原因。
  • L4 管理 Discord Role 映射、第三方集成、密钥和权限策略，并执行 step-up。
  • 价格或礼物变更创建新版本；旧订单、旧礼物请求和历史流水继续使用原快照。
  • 生产配置发布保存配置哈希、版本、操作者、批准链和发布时间，支持回滚到上一版本，但回滚不改写历史业务记录。

12. 关键假设

  • Prototype 只运行在一个 Discord Server、使用人民币单币种，并只接受即时订单。
  • 第三方系统提供用户验证、实时余额、支付和退款事实；若提供原生 hold/capture/release 则优先使用，否则由自有系统维护本地预留。
  • 本地不把第三方真实余额复制成可编辑事实；reservedMinor 只来自有效预留，所有消费入口经过统一业务 API。
  • 示例时区使用 Asia/Shanghai，因为业务币种为人民币；部署前可替换为服务器实际业务时区。
  • 主规格中的订单面板示例采用无畏契约、北美、2 小时和 ¥120，因此示例服务使用 60 分钟单位、最低 2 单位和客户单价 ¥60；陪玩结算价未由业务确认，示例仅用于数据结构与测试。
  • 三种礼物的名称和价格仅为开发数据，未构成经营决定。
  • 语言标签保留在配置结构中，但 P0 自动派单只强制匹配游戏、服务和区服。
