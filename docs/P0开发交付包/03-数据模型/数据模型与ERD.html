Discord Companion Service · Prototype P0

数据模型与实体关系设计

面向统一业务 API 的 PostgreSQL / Prisma 数据模型。覆盖用户绑定、陪玩准入、即时订单、双方就绪、资金预留、取消影响预览、礼物、两类互斥推荐计划、收益与全链路审计。

唯一上位规格：Bot 最小原型设计开发文档 架构：API 独占数据库 金额：minor units + currency 时间：UTC

1. 模型目标

这套模型的重点不是覆盖未来所有陪玩玩法，而是让 P0 的核心闭环在并发、失败重试和人工接管时仍然能对账、追责和恢复。

45Prisma 数据实体
49稳定状态/类型枚举
4后台权限级别
事实来源 PostgreSQL 保存自有业务事实和预留生命周期；第三方系统保存真实用户资金与余额。余额响应固定为 providerBalanceMinor、事件派生的 reservedMinor 与公式所得 availableMinor，后两者绝不持久化。
P0 边界 只支持即时单、单用户、单陪玩、单 Discord Server 和单币种运行。预约/排期、排班、陪玩试音或试听材料、用户选择/指定陪玩、多人订单、多级返佣、自动提现和本地余额账本均不进入 P0。

↔

  • 用户验证

Bot 和 Dashboard 不直接访问 PostgreSQL，也不各自实现价格、状态迁移、金额阈值或权限判断。资金入口统一按 user_id + currency 串行化；幂等作用域为 client + operation + actor + key。

5. 核心数据流

订单提交 reserve

资金锁 → 第三方余额 → 价格快照 → FundReservation/事件 → 订单提交 → 派单 Outbox；不提前 capture。

接单与 readiness

唯一接单 → readiness 截止时间 → 客户确认 + 陪玩确认 → 双方齐备才进入 IN_SERVICE。

礼物请求与批准

请求时 reserve → 客服核对/审批 → 批准后 capture → 消费/返佣 → 播报 Outbox。

订单完成 capture

完成确认 → 捕获预留 → 消费流水 → 陪玩收益 → 计划化返佣 → 释放槽位 → 频道归档。

取消影响预览

版本快照 → 自动处理性 → 预计 release/refund → 处理方式/时效 → 执行时锁内重算。

退款收敛

退款镜像 → 外部成功 → 消费贷方 → 收益调整 → 返佣调整。原始记录不改金额。

互斥推荐归因

计划版本 → 唯一有效来源 → PROMOTER 最多一次 / PLAYER 每个合格来源一次 → 非负调整。

权限同步

source event 防重 → Discord Role 快照 → 四级内部权限 → 撤销旧会话 → 审计。

Bot 配置变更

API staff 授权 → 字段与 Discord 对象校验 → expectedVersion 条件更新 → 当前 JSON + 只追加事件同事务提交 → 刷新 Guild 缓存。

7. 关键约束

风险数据库防线API 防线
一个用户/陪玩同时多单两个 nullable 唯一活跃槽位。状态迁移事务同步占用和释放。
并发订单/礼物超支user_currency_locks 与 append-only 预留事件；来源 XOR/唯一。同一 user_id + currency advisory lock，重读第三方余额后计算 availableMinor。
后台并发覆盖User、PlayerProfile、Order、StaffTask、GiftRequest、FundReservation、ReferralAttribution、Commission、PlayerEarning、ApprovalRequest、OutboxEvent 保存 row_version。以 id + expectedVersion 原子更新并递增；零行返回 409。
IDLE/DND 被错误派单Presence 与准入/可用性组合索引。required_presence=ONLINE；只有 ONLINE 入选。
多人同时抢单订单行锁/条件更新；派单轮次唯一。重新校验候选、状态和活跃槽位。
陪玩单方面开始计费readiness 时间字段与迁移触发器。客户和陪玩都确认才写 SERVICE_STARTED；超时转客服。
取消影响过时CancellationPreview 保存订单/预留版本与过期时间。执行时锁内重算；不一致失效并返回新预览。
目录改价影响旧单版本表和订单完整快照。提交后拒绝更新快照金额。
礼物重复捕获礼物预留、交易幂等键、消费来源唯一。请求时 reserve，批准后 capture；未知结果先查询。
播报失败再次扣款Outbox 独立状态和唯一去重键。只重试播报，不重放扣款用例。
归因/返佣重复有效来源部分唯一、推广者一次性部分唯一、消费来源唯一。两个计划互斥；PLAYER 每个合格来源一次。
归因隐私泄漏敏感字段保存在受限关系列。通用快照、Discord、错误、通知和普通导出只带记录 ID。
退款覆盖原流水退款、消费贷方和两个调整表独立追加。净额由原值减调整合计计算。
冲正方向混乱Adjustment 金额 >= 0；记录只追加。REVERSAL_DEBIT/CORRECTION_DEBIT/CORRECTION_CREDIT 表达方向。
退款状态含义混用Refund 使用四态；原扣款镜像单独保存退款聚合态。成功退款累计后才派生 PARTIALLY_REFUNDED/REFUNDED。
Discord Role 越权/重放四级内部权限；(source, source_event_id) 唯一。API 不相信客户端 Role ID；重复事件无第二次副作用。
Bot 配置越权或并发覆盖guild_id 一行当前态；(guild_id, version) 事件唯一且事件不可改删。API 解析有效 staff actor，逐字段执行 L3/L4 权限；以 expectedVersion 原子更新，冲突返回 CONFIG_VERSION_CONFLICT。
权限继承与高额执行StaffLevel 固定顺序、累积 RolePermission、完整操作者外键。高级别包含全部低级别执行权限；≥ 5000 元最低要求 L4，允许同一人发起并执行，仍强制原因、step-up 与审计。

金额、来源 XOR、readiness、预留累计、归因一致性、部分唯一索引和不可变表更新限制由 PostgreSQL CHECK/触发器补齐。精确 SQL 与事务顺序见同目录《状态枚举与约束.md》。

8. 事务边界

  1. 资金入口：先按 client + operation + actor + key 防重，再持有 user_id + currency advisory lock；每个短事务还锁定 user_currency_locks 行。
  2. 订单：提交时创建预留和事件；接单后分别记录双方 readiness；完成/结案成功时才 capture 并创建消费、收益与返佣。
  3. 礼物：请求创建时 reserve 并创建客服任务；批准后 capture；拒绝、过期或撤回时 release；播报失败只重试 Outbox。
  4. 取消：先创建影响预览；执行时锁订单、预览、资金锁和预留，版本一致并重算相同后才 release/refund 或转客服。
  5. 退款成功：退款与原交易状态、消费贷方、收益和返佣调整同步收敛。
  6. 归因与权限：推荐按被推荐用户串行化；Role 同步按 source event 防重，内部权限固定为 L1–L4。
网络调用边界第三方 API 调用不能占用长数据库事务。advisory lock 跨最终余额读取和外部调用保持；调用前后用短事务登记/收敛。未知结果保持预留，不换幂等键、不提前释放。

9. 更新与删除

只追加 订单事件、预留事件、人工结案、客服备注、审批决定、消费流水、收益/返佣调整、Role 同步事件、Guild Bot 配置事件、风险事件和审计日志。

状态可变 订单、礼物、预留、取消预览、扣款/退款镜像、任务、审批、归因、收益、返佣和 Guild Bot 当前配置只允许改变受控字段；配置必须以版本更新并同步追加事件。

版本化 服务目录、礼物目录、推荐计划、Role 映射和策略配置通过新版本替换。

所有财务和审计相关实体使用 `RESTRICT` 外键并禁止硬删除。纠错使用退款、冲正、禁用、归档或追加更正；L4 同样没有删除财务/审计流水的路径。

10. 实施说明

  • schema.prisma 使用 PostgreSQL provider，并通过 DATABASE_URL 环境变量读取连接地址；不得把真实数据库凭据写入 Schema 或版本库。接入 API 工程后必须执行 prisma validate。
  • 第一批 migration 必须补充《状态枚举与约束.md》列出的 `CHECK`、不可变触发器和 API 数据库角色权限。
  • API 响应的 version 映射 row_version；所有写操作以 id + expectedVersion 单条条件更新并原子递增，禁止“先查版本、后无条件更新”。
  • 余额 DTO 固定为 providerBalanceMinor、reservedMinor、availableMinor、currency、fetchedAt；reserved/available 不建列。
  • 归因受益人、计划、比例、金额和状态不得进入通用快照、Discord、错误、通知或普通导出。
  • 客服任务类型与 OpenAPI 逐值同名：ORDER_ASSIST、CANCELLATION_ASSIST、GIFT_REVIEW、PLAYER_START_LATE、PLAYER_NO_SHOW、CUSTOMER_NO_SHOW、SERVICE_INTERRUPTED、COMPLETION_REVIEW、DISPUTE、AUTOMATION_FAILURE。
  • GuildBotConfig 每个 Guild 一行，Bot 启动/重启时通过共享 API 重建缓存；GuildBotConfigEvent 只追加。配置 JSON 不得保存 Token、数据库/支付/Webhook/OAuth Secret、目录价格、礼物或返佣规则。
  • Bot 配置最终授权在 API：L3 管理日常字段，L4 继承 L3 并管理玩家与 L1–L4 Role 映射；Discord Role、Bot 本地缓存和客户端提交的级别均不能直接授权。
  • 种子数据至少创建 L1–L4 权限矩阵、初始 CNY 金额策略、一个活动服务版本、一个活动礼物版本和单 Server 的 Role/频道配置。
  • 迁移顺序：合成历史已捕获预留；GiftRequest.PAID → CAPTURED；旧 Referral 转 ProgramVersion/Attribution 并回填 Commission；Adjustment 方向显式化；最后启用检查与部分唯一索引。
  • 数据库集成测试重点覆盖：并发订单+礼物、双方 readiness、取消预览失效、capture 未知收敛、推广者第二次奖励拒绝、陪玩逐来源去重和 Role 事件重放。

关键假设

  • 订单提交 reserve、完成 capture；礼物请求 reserve、批准 capture。
  • 订单和礼物各最多一条预留和一条原始 capture 镜像；重试使用同一记录和幂等键。
  • 每个订单一条原始陪玩收益；退款和纠错以调整记录表达。
  • PROMOTER_FIRST_PURCHASE 每个归因最多一笔；PLAYER_LIFETIME 每个合格消费来源最多一笔。
  • 派单配置固定要求 ONLINE；IDLE、DND、OFFLINE、UNKNOWN 均不进入候选池。
  • 两个推荐计划对同一被推荐用户互斥；Commission 只引用 Attribution，不引用泛化 Referral。
  • Prototype 使用 CNY 初始阈值，但每条金额记录仍保存货币代码。
